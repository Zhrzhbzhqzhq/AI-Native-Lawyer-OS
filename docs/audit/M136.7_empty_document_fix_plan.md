# M136.7 Empty Document Fix Plan

Date: 2026-07-09
Target matter: m-mrco9m97-dbygsz

Purpose

Provide a minimal, low-risk plan to prevent or mitigate empty document records like the three observed ("起诉状", "证据清单", "诉讼策略建议"). This is a design/implementation plan only — no code/schema/data changes are made as part of this document.

Problem definition

- Root cause (high probability): the AI intake flow (`POST /intake/ai-create-matter`) calls `AIPipelineService.run()` which returns `steps.documents`; `intakeRoutes.ts` persists each returned document via `documentService.createForMatter(...)` but does not pass `content` (only sets `content_uri: ''`), producing draft document rows with null/empty content.
- Symptom: empty draft documents are visible in the Documents Workspace, increasing counts and creating UX confusion.

Impact scope

- Affected table: `documents` (rows with `content = null` and `content_uri = ''`).
- Affected flows: `POST /intake/ai-create-matter` (primary), other intake confirm endpoints also capable of creating empty docs.
- UX: `Documents Workspace` (`apps/frontend/src/app/matters/[matter_id]/documents/page.tsx`) will list and allow selection of these drafts; selected body displays placeholder `—`.
- Risk level: Low functional risk but medium UX/data-quality concern.

Fix goal

- Prevent creation of persistent empty document stubs by default.
- Keep changes minimal, reversible, and limited to application code (no schema changes).
- Provide short-term UX mitigation while source fix is implemented.

Non-goals

- Do not delete or mutate existing data in DB.
- Do not change DB schema.
- Do not remove AI functionality; prefer to harden persistence logic.

Three candidate solutions (comparison)

A. Backend source fix — only create documents when AI-provided `content` exists

- Description: In `intakeRoutes.ai-create-matter` (the documents persistence loop), require that a document object has non-empty `content` (or meaningful `content_uri`) before calling `documentService.createForMatter(...)`. If `content` absent, skip creation.
- Pros:
  - Stops empty stubs at the source.
  - Keeps DB clean and accurate.
  - Minimal UX changes required.
- Cons:
  - If some business use-cases expect placeholder docs, those would be blocked (requires explicit design change).
  - Slight behavior change for AI pipeline consumers — must be documented.
- Effort: small (single handler change), low-risk, reversible.

B. Backend preserve placeholders but mark explicitly

- Description: Continue creating document rows even if `content` absent, but add a non-schema change metadata pattern (e.g., set `status: 'placeholder'` or include a recognisable `document_type`/title prefix) and ensure logs/audit records note `content_missing=true`.
- Pros:
  - Preserves existing behavior (no upstream contract change).
  - Makes placeholders discoverable and filterable without schema migration.
- Cons:
  - Requires coordination: change to `status` semantics or `document_type` conventions may ripple in UI/business logic.
  - Still leaves empty rows in DB (data quality not improved).
- Effort: small to medium (handler changes + docs + possibly UI awareness).

C. Frontend filtering/display-only mitigation

- Description: Keep backend unchanged. In `apps/frontend/src/app/matters/[matter_id]/documents/page.tsx`, filter out (or visually de-emphasize) documents where both `content` and `content_uri` are empty when rendering the list. Optionally show a toggle to include placeholders.
- Pros:
  - No backend change required — quickest to implement for demo UX.
  - Reversible and non-destructive.
- Cons:
  - Does not fix data quality in DB.
  - Other consumers of the API (integrations, other frontends) still see empty docs.
- Effort: small (client-side filtering and message change).

Recommendation (preferred)

- Implement A (backend source fix) as primary, plus a light frontend UX hint as temporary mitigation.
  - Rationale: fixing at source prevents recurrence and keeps DB clean. Frontend hint reduces immediate demo impact during rollout.
  - Do not change schema. Do not delete existing records in the fix rollout — plan for a separate data-cleanup after business sign-off.

Minimal modification plan (files and changes)

Note: these are minimal code locations to change — this document does not implement the changes.

- Primary backend change (A):
  - File(s): `apps/backend/src/routes/intakeRoutes.ts`
  - Change points:
    - In `ai-create-matter` handler where documents are persisted (the `// Documents` loop), modify the loop to:
      - Extract `content` from AI result item `d` if present.
      - If `content` is non-empty (non-whitespace), call `documentService.createForMatter(...)` with `content` included.
      - Else, skip creating the document (or log and record in `meta.ai.errors` / monitoring).
  - Expected code delta: ~10–20 LOC.

- Frontend light mitigation (optional, short-term):
  - File(s): `apps/frontend/src/app/matters/[matter_id]/documents/page.tsx`
  - Change points:
    - In `fetchDocuments()` or rendering section, filter `arr` by `!( (d.content === null || d.content === '') && (d.content_uri === null || d.content_uri === '') )` before `setDocuments(arr)`.
    - Optionally show an information text when hidden placeholders exist (e.g., "N placeholders are hidden — show" toggle).
  - Expected code delta: ~10–15 LOC.

Alternative (B) minimal changes

- File(s): `apps/backend/src/routes/intakeRoutes.ts`
- Change: when creating documents with empty content, set `status: 'placeholder'` (if accepted) or prefix `title` with `[placeholder]` and add a structured log entry.
- Pros/Cons: as above.

Risk analysis

- Regressions: altering creation behavior (A) may change downstream counts or cause expectations mismatch in any downstream systems that assume a document row will always be created for AI output. Mitigation: add a migration note and coordinate with stakeholders.
- UX: frontend filtering may hide records admin users expect to see. Mitigation: add a toggle to show placeholders.
- Data integrity: source fix avoids further pollution. No existing data is changed by this plan; cleanup is a separate process.

Rollback plan

- Backend change (A) rollback: revert the single commit in `apps/backend/src/routes/intakeRoutes.ts` that adds the `content`-presence guard — deployment rollback to previous version.
- Frontend change rollback: revert the small filtering commit.
- No DB changes are made, so rollbacks are code-only and trivial.

Acceptance criteria (验收)

1. Unit / Integration tests
   - New/updated unit tests cover the `ai-create-matter` documents persistence behavior: when AI returns document objects without `content`, the system does not create document rows (or creates marked placeholders if B chosen).
2. Manual test (happy path)
   - Run the AI intake flow in a staging environment with an AI response that lacks `content` and confirm no empty document rows are written to DB for that matter.
3. UI verification
   - Documents Workspace no longer shows empty placeholder rows in normal view. If frontend toggle used, toggle reveals hidden placeholders.
4. Observability
   - Logs record when AI-run produced candidate documents but none were persisted due to missing content (for monitoring and troubleshooting).

Estimated effort & timeline

- Backend fix (A): 1–2 engineer days (implementation + unit tests + review + deploy to staging).
- Frontend mitigation: 0.5 engineer day (small change + QA).
- QA + rollout: 0.5–1 day.

Notes & follow-ups

- After source fix rollout, schedule a small data-cleanup/triage task (separate change) to decide how to handle existing empty documents (archive, delete, or populate) subject to business approval. That task is intentionally excluded from this plan to preserve read-only constraints.
