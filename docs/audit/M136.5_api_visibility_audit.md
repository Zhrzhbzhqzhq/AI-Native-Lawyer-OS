# M136.5 API Visibility Audit

Audit time: 2026-07-09
Target matter: m-mrco9m97-dbygsz

Objective: Verify whether the 3 empty documents for the target matter are exposed via the API to the frontend Documents Workspace, and whether the frontend will render them (read-only; no code/schema/data changes).

Steps performed (read-only):

1. Inspect backend routes/services that serve documents
   - Files examined:
     - [apps/backend/src/routes/matterRoutes.ts](apps/backend/src/routes/matterRoutes.ts)
     - [apps/backend/src/services/documentService.ts](apps/backend/src/services/documentService.ts)
     - [apps/backend/src/repositories/documentRepository.ts](apps/backend/src/repositories/documentRepository.ts)
   - Findings:
     - `GET /matters/:matter_id/documents` is implemented in `matterRoutes` and calls `documentService.listDocuments(matter_id)` which delegates to `DocumentRepository.listByMatter`.
     - `DocumentRepository.listByMatter` returns `prisma.document.findMany(...)` selecting `content` and `content_uri` and does not filter out empty `content` or `content_uri` values. Therefore the API will return document rows even when `content` and `content_uri` are empty.

2. Confirm database rows (previously exported):
   - The three documents (titles: `起诉状`, `证据清单`, `诉讼策略建议`) were verified in the DB and exported to `docs/audit/M136.4_target_matter_empty_documents_full.csv`.
   - They have `status = 'draft'`, NULL/empty `content` and `content_uri`, and were created at nearly identical timestamps.

3. Inspect frontend Documents Workspace page behavior
   - File examined: [apps/frontend/src/app/matters/[matter_id]/documents/page.tsx](apps/frontend/src/app/matters/[matter_id]/documents/page.tsx)
   - Findings:
     - The frontend calls the backend endpoint at runtime: `fetch(`${base}/matters/${matterId}/documents`)` and assigns the returned array to `documents` without filtering empty content.
     - The UI default-selects the first document in the returned list (if any) and displays `selectedDocument.content ?? selectedDocument.content_uri ?? '—'` for the document body. When both are empty, the UI shows `—` as the body placeholder.
     - The left-hand documents list displays titles and status and will include documents returned by the API.

4. (Optional API simulation) Equivalent DB query used to confirm: 
   - `SELECT ... FROM documents WHERE matter_id='m-mrco9m97-dbygsz'` returned the three target rows (previous read-only DB work already produced these rows).

Conclusions:
- The backend API GET `/matters/:matter_id/documents` returns documents for the target matter including records with empty `content`/`content_uri` (no filtering applied).
- The frontend Documents Workspace performs no further server-side filtering and will render these records in the documents list. The selected document body will display as a placeholder `—` when both `content` and `content_uri` are empty.
- Therefore the three empty documents are visible to the frontend as draft documents (they will increase the documents count and be selectable, but their body appears empty).

Impact on current demo:
- Low functional risk. The UI will show empty draft documents which may look like placeholders; this increases document counts by +3 and may be visually noticeable in the demo but is not blocking.

Recommended next steps (non-destructive):
- If empty placeholder documents are undesirable in demo runs, consider one of the following (requires code/data changes; not performed in this audit):
  - Backend: filter out documents with empty `content`/`content_uri` for the documents list endpoint (or add a query param to include/exclude empties).
  - Frontend: filter documents with empty content when building the documents list or show a clearer placeholder/label (e.g., "placeholder — awaiting content").
  - Or: identify and fix the upstream ingestion/job that created these placeholder documents so they are not created without content.
