# M136.6 Empty Document Source Trace

Audit time: 2026-07-09
Target matter: m-mrco9m97-dbygsz

Objective: determine who/what created the 3 empty documents (titles: 起诉状, 证据清单, 诉讼策略建议) without altering code/data/schema.

Summary conclusion: High confidence the documents were created by the AI intake pipeline via the `ai-create-matter` / `AIPipelineService` → persisted in `intakeRoutes.ai-create-matter` path which calls `documentService.createForMatter(...)` without persisting `content` (only `content_uri: ''`). This produced draft documents with empty `content`/`content_uri` persisted in DB.

Evidence & Findings

1) DB timing correlation (read-only queries performed):
- Documents created in the same second (2026-07-08 22:51:56.*): the three target documents (doc-mrcoa92k-04lifh, doc-mrcoa92n-ukztrc, doc-mrcoa92p-pmhpx4).
- In the same 1-second window there are 8 `evidence` rows created for the same matter (titles: 借款合同/借款合意记录, 银行转账记录, 微信聊天记录, 催收记录, 借款人身份信息, 出借人身份信息, 转账凭证/电子回单, 借款金额确认记录).
- Query used (read-only): counts and row lists for resource types in the time window (performed during audit).
- This simultaneous bulk creation pattern strongly indicates a single pipeline run that persisted multiple resource types (evidence + documents).

2) Suspicious creation path (code):
- Key files inspected:
  - [apps/backend/src/routes/intakeRoutes.ts](apps/backend/src/routes/intakeRoutes.ts)
  - [apps/backend/src/services/ai/AIPipelineService.ts](apps/backend/src/services/ai/AIPipelineService.ts)
  - [apps/backend/src/services/documentService.ts](apps/backend/src/services/documentService.ts)
  - [apps/backend/src/repositories/documentRepository.ts](apps/backend/src/repositories/documentRepository.ts)

- Relevant code snippets (summaries):
  - `AIPipelineService.run(caseSummary)` calls `callStep('documents', ...)` and returns `res.steps.documents` which can be an array of parsed objects like `{ type: '起诉状', content: '...' }` or similar. ([apps/backend/src/services/ai/AIPipelineService.ts])
  - In `intakeRoutes.ai-create-matter` (POST `/intake/ai-create-matter`) the AI pipeline is invoked (`const aiRes = await pipeline.run(caseSummary)`), and the handler persists items from `aiRes.steps.documents` in a loop. Crucially, the persistence call is:

    await documentService.createForMatter(matter_id, { document_id: `doc-...`, title: String(title), document_type: String(document_type), content_uri: '', version: 'v1', status: 'draft' })

    — note: `content` is not passed to `createForMatter`, while `content_uri` is explicitly set to an empty string. ([apps/backend/src/routes/intakeRoutes.ts] around the `// Documents` block)

  - `DocumentRepository.create` (via `DocumentService.createDocument`) sets `content: typeof data.content === 'string' ? data.content : null` and `content_uri: data.content_uri ?? ''`, meaning omitting `content` results in `content = null` and an explicit `content_uri: ''` persists an empty string. ([apps/backend/src/repositories/documentRepository.ts])

3) Other code paths that can create empty docs (also inspected):
- `intakeRoutes.confirm-challenge-document` and `intakeRoutes.confirm-document-update` also call `documentService.createForMatter(...)` with `content_uri: ''` and `status: 'draft'`. These endpoints are user-driven intake confirmation flows and could also create empty documents if invoked with empty content.
- Tests and fixtures: several tests create documents with `content_uri: ''` (e.g., `apps/backend/tests/*`, `apps/backend/tests/intake.test.ts`), meaning codebase patterns use empty `content_uri` in tests; however test data is isolated and unlikely to affect production DB.

Determination of Source:
- Given the tight timestamp clustering across multiple resource types (evidence + documents) and the exact persistence pattern used in `intakeRoutes.ai-create-matter` (persist documents but intentionally not including `content`), the most likely source is the AI intake pipeline run that executed for this matter and wrote evidence + document stubs.
- Confidence: High (correlation of timestamps + matching persistence code path). Not absolute proof (no audit log showing the HTTP request), but sufficient for operational triage and remediation planning.

Is a fix required?
- Severity: Low functional risk (UI shows empty draft documents), but UX/integrity-wise it's undesirable to persist document records without content.
- Recommendation: Yes — fix is recommended to avoid persisting empty document stubs unintentionally.

Suggested non-destructive remediation boundaries (recommendations):
1. Backend change (preferred):
   - Update the AI intake persistence code to persist `content` when available, or avoid creating document rows when `content` is empty unless explicitly requested.
   - Specific patch location: in `apps/backend/src/routes/intakeRoutes.ts` inside the `ai-create-matter` handler where it iterates `for (const d of docs.slice(0, 20)) {...}` — either include `content` from `d` when calling `createForMatter`, or skip creating if `content` is empty and `document_type` is not a placeholder.
   - Rationale: Centralized fix; prevents future empty doc records from AI pipeline.

2. Frontend mitigation (short-term):
   - Filter out documents with both empty `content` and empty `content_uri` from the documents list, or show a clearer placeholder label (e.g., "占位文书（待 AI / 待填充）").
   - Location: `apps/frontend/src/app/matters/[matter_id]/documents/page.tsx` — before assigning `setDocuments(arr)` or when rendering `filteredDocs`, apply filter.
   - Rationale: Non-destructive and immediate UX improvement for demos.

3. Ingestion pipeline / AI pipeline hardening:
   - Ensure `AIPipelineService`'s `callStep('documents')` parse_meta validation handles empty parsed arrays and surface errors; consider returning explicit `{ type, content }` pairs with content validated, and surface `meta.ai.errors` to operators.
   - Rationale: Avoid silent persistence of empty outputs.

4. Operational audit/logging:
   - Add auditing/log lines when `ai-create-matter` persists documents, including matter_id, created document_ids, and whether `content` present; store or stream to logs for traceability.
   - Rationale: makes root-cause tracing definitive in future incidents without DB forensic timing correlation.

- Action: I recommend applying the backend fix (persist content when present; skip persisting empty docs) and adding audit logs. If you want, I can prepare a small PR patch that returns content during persistence or adds a guard—read-only investigation only for now.
