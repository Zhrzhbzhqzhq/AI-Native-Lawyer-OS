# M150.2 Dispatch Engine Technical Design

## 1. Dispatch Engine 产品定位
Dispatch Engine（调度引擎）是 TodayService 背后的规则型决策层（Rule-based Decision Layer）。

其核心职责：
- 基于现有 Matter 业务对象，计算“今天什么最重要、什么在等待律师、什么已完成、下一步建议是什么”。
- 输出可解释（explainable）的调度结果，供 TodayService 聚合并返回给前端。

定位强调：
- 它是“规则判断引擎”，不是 AI 推理引擎。
- 它输出建议，不直接执行业务操作。

## 2. 技术边界
M150.2 仅定义设计，不做实现。

边界内：
- Dispatch Engine 模块拆分
- 输入/输出结构定义
- priority/stage/waiting/completed/nextAction 等规则定义
- 错误与空态策略定义

边界外：
- 不修改数据库 Schema
- 不新增任何表
- 不实现 API
- 不实现 UI
- 不引入外部 AI/LLM 调用
- 不新增消息队列/调度队列

## 3. 不做什么
M150.2 第一版（v1）明确不做：
- 不做实时调度（仅请求时计算）
- 不做自动执行（不会自动创建/修改 Task、Document、Evidence）
- 不做 Workflow 状态迁移
- 不做跨系统依赖（无外部 SaaS、无第三方工作流）
- 不做复杂机器学习评分

## 4. 与 TodayService 的关系
职责分层：
- TodayService：应用层聚合与接口编排（面向 `/today/dashboard`）
- Dispatch Engine：规则决策内核（面向“计算”）

调用关系建议：
1. TodayService 收集原始数据（Matter + 子对象）
2. TodayService 调用 DispatchEngine 计算调度结果
3. TodayService 组装统一响应 DTO

Dispatch Engine 不感知 HTTP、Route、前端字段细节。

## 5. 内部模块建议
建议目录（仅设计）：

```txt
apps/backend/src/services/today/dispatch/
  dispatchEngine.ts
  priorityEngine.ts
  stageEngine.ts
  suggestionEngine.ts
  dashboardAggregator.ts
  dispatchTypes.ts
  dispatchRules.ts
```

### 5.1 DispatchEngine
- 入口模块
- 编排 PriorityEngine / StageEngine / SuggestionEngine
- 产出 Matter 级 DispatchResult

### 5.2 PriorityEngine
- 计算每个 Matter 的 `priorityScore` 与 `priorityLevel`
- 输出可解释的 score breakdown

### 5.3 StageEngine
- 基于对象覆盖率判断案件当前阶段（stage）
- 输出 `stageCode`, `stageLabel`, `stageConfidence`

### 5.4 SuggestionEngine
- 基于阶段和缺口生成 `nextActions`
- 标记 action 的优先级、理由、预计耗时

### 5.5 DashboardAggregator
- 将所有 Matter 的 DispatchResult 聚合为：
  - `completed`
  - `waiting`
  - `nextActions`
  - `activeMatters`
- 负责全局排序与裁剪（limit）

## 6. 输入数据结构
Dispatch Engine 输入必须来自现有表，不新增结构。

```ts
export type DispatchEngineInput = {
  now: string
  matters: Array<{
    matter_id: string
    title: string
    status: string
    updated_at: string
    closed_at?: string | null
    archived_at?: string | null
  }>
  tasksByMatter: Record<string, TaskLite[]>
  materialsByMatter: Record<string, MaterialLite[]>
  evidenceByMatter: Record<string, EvidenceLite[]>
  factsByMatter: Record<string, FactLite[]>
  issuesByMatter: Record<string, IssueLite[]>
  lawsByMatter: Record<string, LawLite[]>
  argumentsByMatter: Record<string, ArgumentLite[]>
  documentsByMatter: Record<string, DocumentLite[]>
  aiRecordsByMatter: Record<string, AiRecordLite[]>
  workspacesByMatter: Record<string, WorkspaceLite[]>
}
```

`Lite` 对象字段最小集：
- 通用：`id`, `status`, `updated_at`, `created_at`
- Task：额外 `priority`, `due_date`, `title`
- Document：额外 `content`, `content_uri`, `title`
- Evidence：额外 `description`, `relevance`, `title`
- AiRecord：额外 `ai_task_type`

## 7. 输出数据结构
```ts
export type DispatchEngineOutput = {
  generatedAt: string
  matterDispatch: MatterDispatchResult[]
  dashboard: {
    completed: CompletedItem[]
    waiting: WaitingItem[]
    nextActions: NextActionItem[]
    activeMatters: ActiveMatterItem[]
  }
  meta: {
    engineVersion: 'rule_v1'
    partial: boolean
    warnings: string[]
  }
}

export type MatterDispatchResult = {
  matterId: string
  matterTitle: string
  stage: {
    code: 'MATERIALS' | 'EVIDENCE' | 'FACTS' | 'ISSUES' | 'LAWS' | 'ARGUMENTS' | 'DOCUMENTS' | 'REVIEW'
    label: string
    confidence: 'low' | 'medium' | 'high'
  }
  priority: {
    score: number
    level: 'P1' | 'P2' | 'P3'
    reasons: string[]
  }
  waiting: WaitingItem[]
  completed: CompletedItem[]
  nextActions: NextActionItem[]
  risks: RiskHint[]
}
```

## 8. priorityScore 规则
### 8.1 评分公式（v1）
每个 Matter 计算分值（0 起）：
- +4：存在逾期任务（`due_date < now` 且任务未完成）
- +3：`waiting` 项数量 >= 3
- +2：存在空内容文书草稿（`document.status='draft'` 且 `content` 空）
- +2：存在弱证据或待核验证据（`status in ['weak','pending']` 或 `description/relevance` 为空）
- +1：最近 24h 无更新且仍有待处理项
- +1：Matter `status='active'` 且 `nextActions` > 0

### 8.2 等级映射
- `score >= 7` -> `P1`
- `score 4..6` -> `P2`
- `score <= 3` -> `P3`

### 8.3 可解释性
必须输出 reasons，例如：
- `OVERDUE_TASK_FOUND`
- `EMPTY_DRAFT_DOCUMENT`
- `WEAK_EVIDENCE_PRESENT`

## 9. stage 判断规则
采用“对象覆盖率 + 缺口优先”判定当前阶段。

判定顺序（从前到后）：
1. `materials > 0 && evidence = 0` -> `EVIDENCE`
2. `evidence > 0 && facts = 0` -> `FACTS`
3. `facts > 0 && issues = 0` -> `ISSUES`
4. `issues > 0 && laws = 0` -> `LAWS`
5. `laws > 0 && arguments = 0` -> `ARGUMENTS`
6. `arguments > 0 && documents = 0` -> `DOCUMENTS`
7. `documents > 0` -> `REVIEW`
8. 否则（几乎无对象）-> `MATERIALS`

confidence 规则（简单）：
- high：命中规则且分母对象差距明显
- medium：命中规则但存在跨阶段对象
- low：对象稀疏或冲突

## 10. waiting 判断规则
纳入 `waiting` 的对象：
- Task：`status in ['pending','active','draft']`
- Document：`status='draft' && (content 为空)`
- Evidence：
  - `status in ['pending','weak']`，或
  - `description` 为空，或 `relevance` 为空
- Issue：`status='draft'`
- Argument：`status='draft'`
- AiRecord：`status in ['pending','waiting','review_required']`（若存在）

waitingReason 映射：
- Task pending -> `待律师执行任务`
- 空草稿文书 -> `待律师补全文书`
- 弱证据 -> `待律师补强证据`
- 争议焦点草稿 -> `待律师确认焦点`

## 11. completed 判断规则
“今日完成”窗口：当日 `00:00:00` 至 `now`。

纳入条件：
- Task：`status in ['done','completed','accepted']`
- Document：`status in ['final','completed','accepted']`
- Evidence：`status='accepted'`
- Fact/Issue/Law/Argument：`status in ['done','completed','accepted']`
- AiRecord：`status in ['ok','done','completed']`

并要求 `updated_at` 在今日窗口内。

## 12. nextAction 生成规则
SuggestionEngine 仅用规则生成，不调用 AI。

规则集（单 Matter 最多 3 条）：
1. `materials>0 && evidence=0` -> 生成证据草稿
2. `evidence>0 && facts=0` -> 生成事实清单
3. `facts>0 && issues=0` -> 提取争议焦点
4. `issues>0 && laws=0` -> 推荐关联法条
5. `laws>0 && arguments=0` -> 生成法律论证
6. `arguments>0 && documents=0` -> 起草文书
7. `documents(draft, empty content)>0` -> 完善文书草稿
8. `overdue tasks > 0` -> 优先处理逾期任务

NextAction 字段建议：
- `actionCode`
- `title`
- `reason`
- `priority`
- `etaMinutes`
- `targetWorkspace`（如 `evidence`, `facts`, `documents`）

## 13. activeMatter 排序规则
筛选：
- Matter `status='active'`
- `archived_at is null`
- `closed_at is null`
- 且满足：`waitingCount>0 || nextActionCount>0 || 最近24h有更新`

排序键：
1. `priorityLevel`（P1 > P2 > P3）
2. `priorityScore`（desc）
3. `overdueTaskCount`（desc）
4. `updatedAt`（desc）

## 14. 风险提示规则
RiskHint 仅做提示，不触发自动动作。

风险项示例：
- `RISK_OVERDUE_TASK`: 存在逾期任务
- `RISK_EMPTY_DRAFT_DOC`: 文书草稿为空
- `RISK_WEAK_EVIDENCE`: 弱证据比例高
- `RISK_STALLED_MATTER`: 48h 无更新且仍有 waiting
- `RISK_NO_NEXT_ACTION`: 有待处理项但无可生成下一步

风险等级建议：
- high / medium / low

## 15. empty state 规则
返回结构稳定：
- `dashboard.completed = []`
- `dashboard.waiting = []`
- `dashboard.nextActions = []`
- `dashboard.activeMatters = []`

并输出：
- `meta.partial = false`
- `meta.warnings` 可含 `NO_ACTIVE_MATTERS`

部分数据源失败时：
- 失败分组置空
- 其余分组正常返回
- `meta.partial = true`
- `meta.warnings` 记录失败来源

## 16. 错误处理规则
策略：fail-soft 优先，fail-fast 兜底。

- 单 Matter 计算失败：
  - 不影响其他 Matter
  - 记录 warning：`MATTER_DISPATCH_FAILED:<matter_id>`
- 单数据源失败（如 tasks）：
  - 该源视为空
  - warning：`TASK_SOURCE_FAILED`
- 核心数据源全失败（matters 不可读）：
  - 返回 500（在 API 层）

日志字段建议：
- `request_id`
- `engine_version`
- `matter_count`
- `partial`
- `elapsed_ms`

## 17. 规则优先级
当规则冲突时，按以下优先级处理：
1. 数据有效性规则（空值/异常防护）
2. 风险规则（逾期、高阻塞）
3. priority 规则
4. stage 规则
5. nextAction 规则
6. 展示排序规则

同一类型 nextAction 去重：
- 按 `actionCode` 去重
- 保留最高优先级版本

## 18. 后续接入 LLM 的边界
M150.2 明确：v1 不接 LLM。

未来（v2）接入边界：
- LLM 只能“增强建议文案与解释”，不能直接改 priorityScore 主规则。
- LLM 输出必须经过规则校验（guardrail）才能展示。
- LLM 失败时应退化为纯规则结果（不影响接口可用性）。

## 19. 与 AI Work Record 的关系
AiRecord（`ai_records`）在 M150.2 的角色：
- 作为“AI 活动历史与状态”信号源
- 参与 completed / waiting / risk 判断

边界：
- 不新增 ai_records 字段
- 不将 Dispatch 结果写回 ai_records
- 不要求 ai_records 成为唯一真值（仍以业务对象状态为主）

建议字段使用：
- `ai_task_type`：标识任务类别（如 conversation / analysis）
- `status`：判断是否等待/完成
- `updated_at`：参与“今日完成”与“停滞风险”判断

## 20. M150.3 / M150.4 / M150.5 拆分建议
- M150.3（Dispatch Engine Implementation）
  - 实现 DispatchEngine + 子引擎
  - 单元测试覆盖 priority/stage/nextAction/waiting/completed
  - 提供可复用内部 DTO

- M150.4（TodayService Integration）
  - TodayService 接入 DispatchEngine
  - 输出 `/today/dashboard` 统一结构
  - 加入 partial/warnings 传递

- M150.5（Frontend & Observability）
  - Today UI 接入新结构
  - 风险提示与空态渲染
  - 指标与日志埋点（耗时、分组数量、partial 比例）
