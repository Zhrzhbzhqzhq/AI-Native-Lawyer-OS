# M161.1 Matter Engine V1 Design

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: 设计 `Matter Engine V1` 的最小实现方案，用于案件阶段推进与对 `Task Engine` 的调用（设计-only；不改代码/Schema/表；不实现 API/UI）。

目标摘要

- Matter Engine 负责在 `Matter` 级别驱动案件阶段（stage）与高阶状态（state），并在需要时调用 `Task Engine`（通过 `TaskService.transitionTaskStatus`）以推进或创建 Task。Today 仍为只读消费方。
- V1 要求最小且可审计：使用现有表（`tasks`/`timelines`/`workflow_events`/`ai_records`），把所有业务逻辑放在后端 Service 层。

## 1. Matter Engine V1 定位

- 责任：在 `Matter` 维度管理案件进度（stage）并作为 Task Engine 的编排层，负责：
  - 监听高阶业务事件（上传证据、法院受理、审判结果、委托变更等）；
  - 决定何时创建 Task，何时触发 Task 状态变更；
  - 产生日志（Timeline / WorkflowEvent）以便审计与 Today 阅读。
- 不做：不直接执行业务侧写回（如提交法院）、不做复杂自动化（V1）。


## 2. Matter Engine 与 Task Engine 的边界

- Matter Engine：高阶规则（案件阶段推进、何时产生工作项、阶段性校验），以 Matter 为粒度决策。
- Task Engine（已实现的 TaskStateMachine + TaskService.transitionTaskStatus）：负责单个 Task 的生命周期校验与状态变更。
- 交互方式：Matter Engine 调用 `TaskService`（应用层接口），或写入 `WorkflowEvent`/`Timeline` 并由 TaskService 根据约定处理。


## 3. Matter Event 列表

- 客户上传材料（client_upload）
- 律师上传材料（lawyer_upload）
- 证据确认（evidence_confirmed）
- AI 生成草稿完成（ai_draft_completed）
- 律师请求修订（lawyer_request_revision）
- 律师批准草稿（lawyer_approve）
- 文书定稿（lawyer_finalize）
- 法院通知/受理（court_notice）
- 重要日期到达（deadline_reached）
- 人工标记阶段推进（manual_advance）
- 任务完成（task_completed）
- 案件关闭（matter_closed）

说明：事件由 UI 或外部系统产生并写入 `WorkflowEvent` / `Timeline`；Matter Engine 会订阅/轮询这些事件并评估规则。


## 4. Matter Stage 列表

- intake (接案)
- discovery / evidence collection (证据收集)
- pleading / drafting (文书起草)
- pre-trial / negotiation (庭前/协商)
- trial / hearing (开庭)
- execution / enforcement (执行)
- closure (结案)

说明：Stage 为业务可读分段，Matter Engine 在 V1 根据事件与 Task 状态来切换 Stage（非强制写回至 Schema 的字段，写回可选用 `Matter.status` 预留字符串）。


## 5. Matter State 列表

- active
- at_risk (如 overdue 或缺关键材料)
- waiting (等待外部输入，如法院/当事人)
- completed (已完成)
- closed (案件关闭)

说明：`Matter.status` 可复用保存此字符串；Matter Engine 应在变更时写 `Timeline` 并可选择 `Matter.status` 更新（Service 层决定是否写回）。


## 6. Matter Event → Task Event 映射

该映射为 V1 的核心规则表（示例）：

- `client_upload` / `lawyer_upload` → if missing Task of type evidence_collection exists -> TaskEvent: `UPLOAD_COMPLETED` (or MATERIAL_ADDED)
- `ai_draft_completed` → map to TaskEvent: `AI_FINISHED` for related Task
- `lawyer_request_revision` → map to TaskEvent: `LAWYER_REVISION`
- `lawyer_approve` → map to TaskEvent: `LAWYER_APPROVED`
- `lawyer_finalize` → map to TaskEvent: `FINALIZED`
- `deadline_reached` (and Task overdue) → create escalation Task or set Task priority, may emit `START` if auto-trigger desired (V1: only flag)
- `task_completed` → map to TaskEvent: `COMPLETE`

实现细节：Matter Engine locates relevant Task by best-effort matching (task.document_id/evidence_id/material_id or title convention). If no Task exists and event semantic requires work, Matter Engine creates Task via `TaskService.createForMatter`.


## 7. 当前 Active Task 规则

- Active Task 定义（V1）：在 Task.status 属于 `ready_to_start`、`ai_working`、`waiting_lawyer`、`revision_requested`、`ai_revising`、`approved` 的非终态时，视为 active。
- Matter Engine `activeTask` 选取策略：
  1. 首先优先选择 status 非 `waiting_materials` 且 priority 高的 Task
  2. 若多个：选择最近 updated_at 的 Task
  3. 若无 Task：返回 null（Matter 处于无活动任务状态）

用途：用于 Matter 首页与 Dispatch 的“当前工作项”栏（Today 只读）。


## 8. 下一阶段判断规则

Matter Engine 在下列情况下判断阶段推进（示例规则，V1 可硬编码）：

- 当当前 Stage 的核心 Task 集全部 `finalized` 或 `completed` 且无高优先 Task -> 进入下一 Stage
- 若存在 `waiting_materials` 的 Task 且无 pending upload -> Matter 状态标记为 `waiting`
- 若关键 Task overdue（根据 due_date）超过阈值 -> Matter 状态标记为 `at_risk`
- 法院通知（`court_notice`）或重要外部事件 -> 跳转到相应 Stage（如受理 -> trial）

说明：Stage 推进后，Matter Engine 可创建进入下一阶段需要的初始 Tasks（见第9）。


## 9. 何时创建 Task

Matter Engine 在以下场景创建 Task：

- 事件触发且不存在语义对应 Task（例如 `client_upload` 触发但无 evidence collection Task）——创建 evidence_collection Task
- Stage 切换时需要启动的标准 Work Items（例如 trial stage 切换时创建 hearing_preparation Task）
- deadline/escalation 策略触发（创建优先级提升或验证性 Task）

创建行为：使用 `TaskService.createForMatter` 并写入初始 `Timeline` 與 `WorkflowEvent`（payload 包含 reason 与 refs）。


## 10. 何时只推进 Task 状态

当事件只是指示已完成某一步（例如 AI 完成草稿、上传附件完成），且对应 Task 已存在時，Matter Engine 只會调用 `TaskService.transitionTaskStatus(task_id, mappedEvent)` 来推进状态，不會创建新 Task。

原则：优先推进现有 Task；只有在无法找到合适 Task 时才创建新 Task。


## 11. 何时等待律师

Matter Engine 标记等待律师（Matter 状态或 Task 状态为 `waiting_lawyer` / `waiting_materials`）的时机：

- AI 產生草稿後，若该 Task 进入 `waiting_lawyer`，Matter Engine 将不会继续自动推进同 Task，直到律师执行触发（LAWYER_APPROVED / LAWYER_REVISION / FINALIZED）。
- 若缺关键材料，Matter Engine 创建或更新 Task 到 `waiting_materials` 并向 Timeline 写入 `waitingReason`（如需客户上传证据），Today 将显示為 waiting。

实现：Matter Engine 在處理事件后應写入 Timeline（事件类型：`matter:wait_for_lawyer`）并可选择通知（V1 不做实时推送）。


## 12. 何时进入 completed

案件/阶段进入 `completed` 的条件（V1）：

- 对于 Stage：当 Stage 下所有必需 Task 都为 `finalized` 或 `completed` 且无 pending waiting items，则该 Stage 可標记为 completed；Matter Engine 写 Timeline 并可更新 `Matter.status` 为 `active`/下阶段或 `completed`（若最后阶段）。
- 对于 Matter：当所有 Stages 完成且所有关键 Tasks 完成并经过律师定稿/提交，Matter 进入 `closed`/`completed`；写 Timeline 记录闭环事件。

注意：`completed` 与 `finalized` 的区别保持一致：`finalized` 是单个 Task 的“定稿”；Matter `completed` 表示业务流完成。


## 13. 不改 Schema 的 V1 实现方式

实现策略（最小改动）：

1. 在后端新增 `MatterEngine` 模块（Service 层）实现规则引擎（同步函数 + idempotent handlers），不改 Prisma Schema。
2. 事件入口：由后端路由接收 `WorkflowEvent`/UI 触发，或在后台轮询 `workflow_events` 表并处理未消费事件（V1 推荐轮询/cron，避免新增 API）。
3. 对 Task 的操作：调用 `TaskService.createForMatter` 和 `TaskService.transitionTaskStatus`，不要直接写入 TaskRepository。
4. 写审计：所有 Matter 决策写入 `Timeline`（event_type 如 `matter:stage_change`、`matter:task_created`）。
5. 保持幂等：根据 `workflow_event.workflow_event_id` 去重处理，避免重复创建/推进。

实现步骤（短期）：

- 实现 `apps/backend/src/services/matterEngine.ts`（内部规则表 + event handler）
- 在 `MatterService` 或独立调度任务中调用 `matterEngine.handleEvent(workflow_event)`
- 提供管理命令/脚本用于回放历史 `workflow_events`（backfill）


## 14. 暂不做内容（V1 明确排除）

- 不接 LLM（直接调用模型）
- 不自动生成并定稿文书（AI 只能产出草稿，律师必须 Finalize）
- 不新增 API（尽量复用现有 endpoints / 后端轮询）
- 不新增数据库表
- 不做实时推送（通知机制留为 V2）


## 15. M161.2 实现建议

- 分阶段交付：
  1. Phase 1（V1）：实现 MatterEngine 核心规则、事件轮询、Task 创建/推进调用与 Timeline 写入；覆盖关键用例的集成测试。
  2. Phase 2：引入管理界面或调试 API，允許人工回放事件、强制推进或回滚（受权限限制）。
  3. Phase 3：将事件处理从轮询迁移到事件驱动（Webhooks, 消息队列），并安全接入 AI Runtime 提示（非自动执行）。

- 测试策略：端到端场景测试（事件注入 -> MatterEngine -> TaskService 调用 -> Timeline 断言）。
- 监控：记录每个事件处理时长、失败率与被创建 Task 的计数。


## 16. 验收标准

1. MatterEngine 能根据 `WorkflowEvent`（或轮询）创建 Task 并调用 `TaskService.transitionTaskStatus` 完成合理推进（覆盖至少：upload→ready→start→ai_finished→waiting_lawyer→lawyer_finalize→completed 场景）。
2. 所有决定被写入 `Timeline`（含 from/to/actor/reason/refs），且能被 Today 读取并展现为 waiting/nextActions/activeMatters 聚合（只读）。
3. 幂等性：重复提交相同 `workflow_event_id` 不会重复创建 Task 或重复推进状态。
4. 无 Schema 更改，后端构建与项目检查通过。
