# P2.4 Daily Operating System

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: 设计文档 — 定义律师每天使用 LawDesk 的工作节奏与 Today 工作台职责。严格为设计-only：不修改代码、不改 DB schema、不新增表、不实现 API 或 UI。

目的摘要

LawDesk 的目标不是展示统计仪表盘，而是每天告诉律师“今天最该处理的一件事”。Daily Operating System（简称 DOS）定义从早晨打开 Today，到处理 Waiting 项、进入 Matter、完成审核与定稿的日常工作节奏与交互约定，保证 AI-生成化工作不会抢占律师决策权，并使 Today 成为可执行的工作入口（跳转到 Matter/Workspace），而非被动案件列表。

**核心原则（复用 P2.1 / P2.2 / P2.3 要点）**
- 律师驾驶，AI 协助：律师负责判断、批准、定稿，AI 负责加工、草稿与建议。
- Today 是调度层而非真值层：不写业务对象，仅聚合 Matter/Work Item 状态产生可操作建议。
- 首页第一屏必须回答：今天我先做什么？（首要任务明确、直达入口）
- 最小惊扰原则：优先展示最紧要项，避免将所有未决项无差别罗列。

---

**1. Daily Operating System 的定位**

DOS 是面向律师的“每天工作节奏指导手册 + 系统化调度视图”。它：
- 把 `TodayService` / `Dispatch Engine` 的规则输出转为可执行的工作流（Next Action、Waiting、Active Matters、Completed）;
- 强调“单一首要任务”优先级，辅以短列表的 Next Actions;
- 提供快速进入 Matter / Workspace 的路径以在上下文中完成工作；
- 保持可解释性、可审计与律师主导的确认点。

目标用户：独立律师及小型律所日常个体或小组工作。V1 关注个人化、确定性与可回溯。

---

**2. 律师一天的主循环（High-level loop）**

早晨（Open Today）→ 系统给出“一件事”（Top Action）与若干 Next Actions → 律师选择 Top Action 或按优先级工作 → 进入对应 Matter/Workspace 进行处理 → AI 辅助生成/分析（如触发）→ 律师 Review/Request Revision/Approve/Finalize → 完成记录回 Today（Completed）→ 今日循环继续。

细化步骤：
- 打开 Today（自动刷新或按需刷新）
- 浏览首屏 Top Action（单条）与 3~5 条 Next Actions
- 快速决定：执行 Top Action 或选择其他 Next Action
- 跳转到 Matter → 执行律师动作（Start / Add Materials / Approve / Request Revision / Finalize）
- 若触发 AI 生成，等待 AI 完成并进入 `waiting_lawyer`，律师审阅
- 审核完成后，执行定稿或标记已完成
- 返回 Today，继续下一个优先项

---

**3. Morning Brief / 今日工作（Morning Brief）**

Morning Brief 是 Today 首屏的语义：对律师以一天为单位的工作摘要与建议，包含：
- 今日首要任务（Top Action）—— 一个明确的、可跳转的工作项，回答“今天我先做什么？”
- 今日简短目标（3 条短句，例：处理逾期送达，完善起诉状草稿，复核证据链）
- 时间预算提示（预计耗时：短/中/长；或 ETA 分钟区间）
- 风险提示（若有 high-risk 项目，显示简短原因）

生成原则：
- Top Action 由 Dispatch Engine 的 priority 规则＋等待时间权重决定；若存在逾期 Task 或空文书草稿，则优先显示；
- 若多项并列，选择能在 30~90 分内完成的工作优先展示（降低心理阻力）；
- Morning Brief 明确可完成的“今日目标”，并支持律师一键接受并进入处理。

---

**4. Today Dashboard 的职责**

Today Dashboard 必须：
- 呈现四大分组：`Top Action`（首要任务）、`Next Actions`、`Waiting`、`Active Matters` 与 `Completed`（当天完成）；
- 强调首屏 Top Action，Next Actions 作为备选；
- 每个 Waiting 项显示 concise `waitingReason` 与可跳转的目标（Workspace 路径）；
- 支持按优先级/风险快速筛选（P1/P2/P3）；
- 不展示 Demo/Mock 内容（过滤测试数据）；
- 提供可解释信息（为什么是 Top Action，score/reasons 简述）；
- 只读聚合：不在 Today 中修改业务对象，所有写操作回到相应 Matter Workspace 完成。

---

**5. Waiting for Lawyer 的优先级（如何排序）**

Waiting 列表排序优先级（高 → 低）：
1. 高风险且逾期（Overdue + High Risk）
2. 空文书草稿（影响提交的关键文书）
3. 影响诉讼期限的 Task（即将到期或已过期）
4. AI 等待 Review（高置信度低影响的 AI 产物可排在后）
5. 其他（例如弱证据需补强）

次级排序：priorityScore、updatedAt（越久未处理越靠前）、matter priority level（P1/P2/P3）。

显示要点：每个 Waiting 项显示 `短理由` + `预计耗时`（若可估）+ `跳转链接`。

---

**6. AI Today Completed 的呈现方式**

Completed（今日已完成）用于给律师提供闭环感与审计：
- 只展示今日时段（00:00 至 now）内变更为 `finalized` / `completed` 的项；
- 展示最少信息：type、title、matter、完成时间、简短来源（如 AI-generated / Manual）;
- 对于 AI 参与的完成项，显示 AI 元数据（`revision_round`、`confidence_score`、`engine_version`）以便溯源；
- Completed 不占首屏优先位置，但应易于展开查看当日工作量。

---

**7. Next Actions 的呈现方式**

Next Actions 提供明确、可执行的建议：
- 每条包含：`action title`、`理由`（Why）、`priority`、`预计耗时`、`目标 Workspace`（e.g., documents/evidence）
- 限制每个 Matter 最多 3 条建议，全局限制默认 20 条；
- 对建议提供快速动作：`开始`（Start triggers AI 或打开编辑器）、`跳转`（直接到对应 Workspace）、`稍后`（将该项推入 Today 的延后队列）
- Next Actions 为建议，不应自动触发任何写操作，律师需显式 Start 或授权。

---

**8. Active Matters 的呈现方式**

Active Matters 列出今日需要关注的案件卡片（按 priority 排序），卡片包含：
- Matter 标题、stageHint（当前阶段）、waitingCount、nextActionCount、priorityLevel、最后更新时间
- 点击卡片可展开：Top Waiting Items + Top NextActions（便于快速定位）
- 卡片应突出 P1 信号与风险提示（红色/三角等视觉，但在 V1 仅文字说明）

---

**9. 从 Today 进入 Matter 的路径**

- 所有 Today 项目需提供明确 `target` 链接，跳转到对应 Matter 的具体 Workspace 或条目。
- 设计约定：
  - Waiting 文书 -> 跳转到 `Matter / Documents / [document_id] / edit`
  - Waiting 证据 -> 跳转到 `Matter / Evidence / [evidence_id] / detail`
  - Next Action: 生成事实清单 -> 跳转到 `Matter / Facts / create` 或 `Matter / Research`
- 跳转后保持上下文（返回 Today 时保持原来位置），并在跳转入口带上 `referrer=today` 参数以用于审计与返回定位。

---

**10. 从 Matter 进入具体 Workspace 的路径**

- Matter Workspace 作为单案件工作空间，包含 Tabs：Overview / Materials / Evidence / Documents / Research / Execution / Timeline / AI Workspace 等。
- 约定快捷入口：
  - `Start` / `Next Action` → 直接打开对应 Tab 的编辑器或创建界面
  - `Approve` / `Finalize` → 在 Documents Tab 的草稿页执行（并记录审计）
- 保证按需传参：`work_item_id`, `action=review|start|finalize`, `referrer=today`。

---

**11. 律师处理一个等待项的完整流程（端到端）**

示例：律师处理“空文书草稿（待律师补全）”的端到端流程：
1. Today 显示 Top Action：`完善起诉状草稿`（waitingReason: 空内容）
2. 律师点击 Top Action → 跳转至 `Matter / Documents / [docId] / edit`（带 referrer）
3. 律师阅读 AI 生成的 `change_summary` 与 `self_check_report`（若存在）
4. 律师选择：
   - 直接编辑并 `Finalize` → 文档变为 `finalized`，Today 中该项进入 `Completed`
   - 或选择 `Request Revision`：提交 `revision_instructions`（结构化或文本）并保存；系统记录 `revision_round++`，状态转为 `revision_requested`
   - 或 `Add Materials`：上传关键材料，Work Item 状态转为 `ready_to_start`，可手动 Start 或等待系统 Next Action
5. 若律师选择 `Request Revision`：AI 执行 `ai_revising` 并 `Submit for Review` → 返回 `waiting_lawyer` 并在 Today 更新
6. 重复修订直到律师 `Approve` 或 `Finalize`
7. 完成后，系统记录审计条目（who/when/why/engineMetadata），并将该条目纳入今日 `Completed` 聚合

要求：每一步都保留审计痕迹与可回溯的 `revision_round`。

---

**12. 多轮审核的一天内节奏**

目标：让多轮审核在同一工作日内尽量完成，避免跨日“半成品”堆积。

建议节奏与策略：
- 早晨优先处理可在 30~90 分内完成的 Top Action，若一个 Work Item 预计多轮修订（>2 轮），考虑将其排入当日中段并明确时间窗（例如 2 小时）
- 对于 `revision_requested` 的项：AI 轮次应被优先放在律师当天可接受的短窗口内（若 AI 需要外部调用或耗时较长，应在 Today 标注 ETA）
- 限制软上限：默认 5 轮修订提示（超出提示人工处理或分配负责人）
- 当天内若存在多轮交互，系统应在 Today 中将该条目标注为“进行中（多轮）”并显示当前轮次与上次更新时间

提示：若多轮审核频繁占用当天工作时间，系统应提供“暂停并稍后继续”选项，将该项临时移出 Top Action，避免打断短任务的连续完成。

---

**13. 如何避免信息墙（Information Walls）**

信息墙＝律师无法从 Today/Workspace 快速得到决策所需的最小上下文。避免措施：
- 每个 Waiting / NextAction 必须带最小上下文包：`title`, `shortSummary`（2 行内）, `lastUpdated`, `waitingReason`, `requiredInputs`（列举最少 1~3 项）
- 支持“一键预览”在 Today 内展示关键摘录（例如文书首段、证据关键条目）而无需加载完整 Workspace
- 在跳转时携带 `referrer=today` 并在 Workspace 顶部展示 Today 摘要卡，方便律师确认来由
- 优先展示可立即执行的信息（预计耗时、责任人、影响范围），避免隐藏在多级菜单下

---

**14. V1 不做什么（明确边界）**

V1 明确不实现：
- 在 Today 中做任何写回业务对象的自动化（不自动执行 Approve/Finalize/Batch operations）
- 复杂 ML 驱动的优先级自学习（仅使用可解释规则）
- 增加数据库表或改变现有 schema
- 引入外部任务队列或自动执行器
- 将 Today 变成实时 Workflow 引擎

---

**15. V2 扩展方向**

后续可选能力（需严格审计与权限控管）：
- 半自动化授权执行：律师可配置授权范围，在授权内允许系统或经授权的 AI 批量执行某类建议（可回滚）
- 学习驱动优先级：基于历史完成时间与成功率微调 priorityScore
- SLA 管理与自动提醒（逾期自动上报）
- 团队协作视图（分配给同事、外部协作人）与交接箱
- 更丰富的可视化时间块（日程化 Today Actions，与日历集成）
- 可配置规则面板（让律所管理员调优 rule weights）

---

**16. 与 P2.5 Work Item Specification 的关系**

- P2.5（建议）负责定义具体 Work Item 的 Schema、元数据（如 `revision_round`, `waitingReason`, `requiredInputs`）与事件日志格式；
- DOS（P2.4）依赖 P2.5 提供稳定的 Work Item 表述以便 Today 调度与界面展示；
- P2.4 中定义的字段需求（例如 `requiredInputs`, `estimatedMinutes`, `revision_round`）应在 P2.5 中给出明确字段定义与审计要求；
- 实施顺序：P2.5 定义 -> P2.4 使用 -> M150.* 基于现有字段实现 Today 聚合（V1 遵守不改 Schema 的约束）。

文档交付验收必须满足：

4. 明确 Waiting for Lawyer 的排序规则与显示字段（第5节）
5. 规定 AI Completed、Next Actions、Active Matters 的呈现方式（第6、7、8节）
6. 指定从 Today 到 Matter 的跳转路径与入参约定（第9、10节）
7. 给出律师处理单个等待项的完整端到端流程与审计点（第11节）
8. 定义一天内多轮审核的节奏与软上限（第12节）
9. 给出信息墙规避策略（第13节）
10. 明确 V1 不做事项与 V2 扩展方向（第14、15节）
11. 列示与 P2.5 的依赖关系（第16节）
12. 提供若干可执行的验收测试用例（示例）：
    - 场景 A（空文书草稿）: 创建 draft document (content empty) -> Today waiting 显示 `待律师补全文书内容` -> 跳转编辑 -> 律师 Finalize -> Today Completed 显示
    - 场景 B（逾期 Task）: 逾期 Task 出现在 Top Action -> 律师跳转并完成 -> Completed 被记录
    - 场景 C（AI 修订多轮）: 律师 Request Revision -> AI revisit -> 多轮后 Approve -> revision_round incremented & audit logged

交付要求：仅新增文档文件，不修改代码或数据库；文档需能指导后续 M150.2 / M150.3 的实现。
