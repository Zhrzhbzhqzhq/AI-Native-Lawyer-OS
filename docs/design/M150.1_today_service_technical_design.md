# M150.1 TodayService Technical Design

## 1. 产品定位
Today（今日工作台，Today Workspace）在 V2 中定位为 AI Dispatch Center（AI 调度中心）前端入口，而不是传统 Dashboard。

TodayService 的职责是：
- 聚合已有 Matter 业务对象状态，输出“今天 AI 与律师分别该做什么”。
- 生成 Today 页面所需四个核心分组：
  - `completed`
  - `waiting`
  - `nextActions`
  - `activeMatters`

TodayService 不承担业务真值存储，业务真值仍在 Matter 及其子对象。

## 2. 技术边界
M150.1 仅定义后端聚合层设计，不做实现。

边界内：
- 新增聚合服务设计：TodayService（应用层）
- 新增读取型接口设计：`GET /today/dashboard`
- 定义返回类型、规则、错误处理

边界外：
- 不改数据库 Schema
- 不新增数据库表
- 不实现后端代码
- 不实现前端 UI
- 不调整现有 Matter Workspace 业务流程

## 3. 不做什么
M150.1 明确不做：
- 不新增持久化“today_items”或“dispatch_queue”表
- 不在 Today 中引入新的业务状态机
- 不把 Today 变成 Workflow 执行引擎
- 不改变任何既有路由行为（如 `/matters/:id/*`）
- 不做 AI 模型调用（先基于规则聚合）

## 4. 数据来源
仅使用现有对象与现有数据：
- Matter（`matters`）
- Material（`materials`）
- Evidence（`evidence`）
- Fact（`facts`）
- Issue（`issues`）
- Law（`laws`）
- Argument（`arguments`）
- Document（`documents`）
- Task（`tasks`）
- AI Work Records（`ai_records`，Prisma model: `AiRecord`）
- Workspace（`workspaces`）

建议的核心字段（最小可用）：
- Matter: `matter_id`, `title`, `status`, `updated_at`, `archived_at`, `closed_at`
- Task: `task_id`, `matter_id`, `title`, `status`, `priority`, `due_date`, `updated_at`
- Evidence/Fact/Issue/Law/Argument/Document: `status`, `updated_at`, `matter_id`
- AiRecord: `ai_task_type`, `status`, `matter_id`, `created_at`, `updated_at`

## 6. GET /today/dashboard API 设计
- Method: `GET`
- Path: `/today/dashboard`
- Query（可选）:
  - `limit`：默认 `20`，限制 `activeMatters` 返回数量
  - `matter_status`：可选过滤（例如 `active`）
  - `include_archived`：默认 `false`

响应语义：
- 读取型聚合接口，不写入任何业务对象。
- 返回 Today 所需完整聚合结果，不要求前端二次拼装核心逻辑。

## 8. completed 聚合规则
规则目标：展示“今日已完成”而不是“历史总完成”。

时间窗口：
- 默认按服务端时区“今日 00:00:00 到现在”。

纳入条件（任一命中）：
- Task: `status in ['done', 'completed', 'accepted']` 且 `updated_at` 在今日。
- Document: `status in ['final', 'completed', 'accepted']` 且 `updated_at` 在今日。
- Evidence: `status = 'accepted'` 且 `updated_at` 在今日。
- Issue/Law/Argument/Fact: `status in ['completed', 'accepted', 'done']` 且 `updated_at` 在今日。
- AiRecord: `status in ['ok', 'done', 'completed']` 且 `updated_at` 在今日。

去重策略：
- key: `type + id`。

排序：
- `completedAt desc`。

## 9. waiting 聚合规则
规则目标：展示“AI 等待律师确认/输入”的阻塞点。

纳入条件（优先级从高到低）：
- Task: `status in ['pending', 'active', 'draft']`。
- Document: `status = 'draft'` 且 (`content is null` 或 `content = ''`)。
- Evidence: `status in ['pending', 'weak']` 或 (`description = ''` 或 `relevance = ''`)。
- Issue/Argument: `status = 'draft'`。
- AiRecord: `status in ['pending', 'waiting', 'review_required']`（若存在）。

waitingReason 映射示例：
- 空文书草稿 -> `待律师补全文书内容`
- 弱证据 -> `待律师补强证据说明/关联`
- 草稿争议焦点 -> `待律师确认争议焦点`

排序：
- 优先按 `priority`（若有），再按 `updated_at asc`（越久未处理越靠前）。

## 10. nextActions 聚合规则
规则目标：给出“下一步动作建议”，不自动执行。

v1 使用规则引擎（rule_engine_v1）：
- 若 `materials > 0 && evidence = 0` -> `action: 生成证据草稿`
- 若 `evidence > 0 && facts = 0` -> `action: 生成事实清单`
- 若 `facts > 0 && issues = 0` -> `action: 识别争议焦点`
- 若 `issues > 0 && laws = 0` -> `action: 推荐相关法条`
- 若 `laws > 0 && arguments = 0` -> `action: 生成法律论证`
- 若 `arguments > 0 && documents = 0` -> `action: 起草文书`
- 若 `documents draft > 0 && empty_content > 0` -> `action: 完善文书草稿内容`

输出上限：
- 全局最多 20 条
- 单 Matter 最多 3 条

## 11. activeMatters 聚合规则
规则目标：列出“当前真正活跃且需处理”的 Matter。

筛选条件：
- `matter.status = 'active'`
- `archived_at is null`
- `closed_at is null`

活跃信号（至少一项）：
- 今日有 `waiting` 项
- 今日有 `nextActions` 项
- 今日有对象 `updated_at` 更新（Task/Document/Evidence/Fact/Issue/Law/Argument/AiRecord）

排序：
- `priority`（P1 > P2 > P3）
- `updatedAt desc`

## 12. priority 计算规则（简单规则）
先用可解释、可复核的简单规则：

每个 Matter 计算 `score`：
- +4: 存在逾期 Task（`due_date < now && status not done`）
- +3: `waiting` 项 >= 3
- +2: 存在空内容 draft Document
- +2: 存在 `weak/pending` Evidence
- +1: 最近 24h 无更新且仍有待处理项

映射：
- `score >= 7` -> `P1`
- `score 4..6` -> `P2`
- `score <= 3` -> `P3`

## 13. empty state 规则
接口必须返回结构稳定的空态，不返回 `null` 顶层分组。

当无数据时：
- `completed = []`
- `waiting = []`
- `nextActions = []`
- `activeMatters = []`
- `summary.*Count = 0`
- `meta.partial = false`
- `meta.warnings` 可包含 `NO_ACTIVE_MATTERS`

当部分数据源失败时：
- 能返回的分组照常返回
- 失败分组返回空数组
- `meta.partial = true`
- `meta.warnings` 记录失败来源（例如 `TASK_QUERY_FAILED`）

## 14. 错误处理规则
HTTP 级别：
- `200`: 正常或部分成功（通过 `meta.partial` 区分）
- `500`: 全量失败（例如数据库不可用）

服务内规则：
- 单数据源失败不应拖垮全接口（fail-soft）
- 对异常仅返回通用错误，不泄露内部堆栈
- 统一日志字段：`request_id`, `source`, `elapsed_ms`, `partial`

## 15. 与现有 Matter Workspace 的关系
- Matter Workspace 仍是单 Matter 的业务操作入口（对象级管理）。
- Today 是跨 Matter 的调度视图（调度级聚合）。
- Today 不替代 `/matters/:matter_id/*` 现有路由。
- Today 的动作项点击后应回跳到对应 Matter Workspace/子页执行（例如 `evidence`, `documents`, `facts`）。

## 16. 后续 M150.2 / M150.3 / M150.4 拆分建议
- M150.2（Backend Implementation）
  - 实现 `todayService.ts`、`todayRules.ts`、`todayDataLoader.ts`
  - 新增 `GET /today/dashboard` 只读路由
  - 增加服务级单测（规则与优先级）

- M150.3（Frontend Integration）
  - Today 页面读取 `/today/dashboard`
  - 替换当前 matters 页面中静态“AI 正在推进/等待/完成”块
  - 加入空态与部分失败提示

- M150.4（Quality & Observability）
  - 增加端到端回归用例
  - 接口耗时与错误监控
  - 规则可配置化（仍不改 Schema，先配置文件）

## 17. 验收标准
M150.1 设计完成标准：
- 有独立技术设计文档，明确 TodayService 是聚合层，不是新业务真值层。
- 明确“只用现有数据、不新增表、不改 Schema”。
- 完整覆盖以下设计项：
  - `completed` / `waiting` / `nextActions` / `activeMatters` 规则
  - `priority` 简单规则
  - API 合同与 JSON 类型
  - empty state 与错误处理
  - 与 Matter Workspace 的职责边界
  - M150.2~M150.4 拆分路径
- 文档可直接指导后续实现而无需新增架构假设。
