# M160.2 Task State Machine Design

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: 基于 M160.1 审计结论与 P2.2/P2.5 规范，定义在 V1（不改 Schema、不新增表）的 Task / Work Item 状态机与事件体系，供 Service 层实现与 Today/Dispatch 读取。

目标原则

- 使用现有 `Task` 模型作为 Work Item 最小单元。
- 在 Service 层实现状态机与事件处理逻辑（不改 Schema、不新增表）。
- 记录所有状态变更与触发事件到现有审计表：优先复用 `ai_records`、`timelines`、`workflow_events`，并在 Service 层保证一致性与可追溯性。
- 明确哪些事件由律师触发、哪些可由 AI 触发、以及非法流转规则。

## 1. Task V1 状态列表

使用 P2.2 / P2.5 统一枚举（在 Service 层约定字符串常量）：

- waiting_materials
- ready_to_start
- ai_working
- waiting_lawyer
- revision_requested
- ai_revising
- approved
- finalized
- completed
- error  （内部错误子态，用于标记失败）

说明：`error` 为可选子态，用于记录 AI 或执行时发生的失败；Service 层应把其映射为可读的 `waiting` 警告。

## 2. Task 事件列表

事件分为两类：Command（外部触发，律师或系统/AI 发起）与 Internal（内部/系统自动或 Service 层校验产生）。

Command 事件（外部可触发）
- upload_materials {actor, details}
- start_task {actor, mode: 'ai'|'human'}
- ai_generate_started {ai_record_id}
- ai_generate_completed {ai_record_id, summary}
- lawyer_submit_review {actor}
- lawyer_approve {actor}
- lawyer_request_revision {actor, instructions}
- lawyer_finalize {actor}
- mark_completed {actor}
- assign_owner {actor, owner_user_id}
- add_waiting_reason {actor, reason}
- clear_waiting_reason {actor}
- update_estimate {actor, minutes}
- force_transition {actor, to_state, reason}  // admin/debug only

Internal / System events
- inferred_ready {reason}  // Service 判断材料齐备
- ai_revision_started {ai_record_id}
- ai_revision_completed {ai_record_id}
- ai_error {ai_record_id, code, message}
- timeout_escalation {policy_id}
- auto_recover {reason}

事件应包含：who（actor）/when（timestamp）/why（reason）/refs（task_id、ai_record_id、timeline_id）

## 3. 状态流转表

表格：当前状态 -> 触发事件 -> 新状态 -> 备注

- waiting_materials
  - upload_materials -> inferred_ready OR ready_to_start (如果 Service 校验通过材料齐备)；备注：需要触发 `timeline` 记录。
  - force_transition -> 任意（仅 admin）

- ready_to_start
  - start_task (mode=ai) -> ai_working（创建 AiRecord，记录 ai_record_id）
  - start_task (mode=human) -> waiting_lawyer OR remain ready_to_start（视业务）
  - upload_materials -> remain ready_to_start
  - ready_to_start -> waiting_materials (if material invalidated)

- ai_working
  - ai_generate_completed -> waiting_lawyer（attach ai_record）
  - ai_error -> waiting_lawyer 或 error（取决错误严重性，并记录 ai_record）
  - lawyer_request_revision -> revision_requested（律师在 AI 运行期提出意见）

- waiting_lawyer
  - lawyer_submit_review -> approved OR revision_requested（可直接 approve 或 request revision）
  - lawyer_request_revision -> revision_requested
  - start_task(mode=ai) -> ai_working (再次触发 AI)
  - upload_materials -> ready_to_start (若材料影响决定，或可保持 waiting_lawyer with updated reason)

- revision_requested
  - ai_revision_started -> ai_revising
  - lawyer_add_instructions -> revision_requested (保持，更新指令)

- ai_revising
  - ai_revision_completed -> waiting_lawyer
  - ai_error -> waiting_lawyer 或 error

- approved
  - lawyer_finalize -> finalized
  - upload_materials -> approved (可补材料，但需记录)

- finalized
  - mark_completed -> completed
  - force_transition -> completed (admin)

- completed
  - （终态，通常不再流转）

- error
  - auto_recover / upload_materials / force_transition -> 上一合法状态或 waiting_lawyer

注意：所有流转必须在 Service 层产生一条 Timeline / WorkflowEvent 记录，并在可能时关联 AiRecord。

## 4. 哪些事件由律师触发

- upload_materials
- start_task (mode=human)
- lawyer_submit_review
- lawyer_approve
- lawyer_request_revision
- lawyer_finalize
- mark_completed
- assign_owner
- add_waiting_reason / clear_waiting_reason
- update_estimate

律所用户触发的事件都必须被记录到 `timeline` 以及对应的 `workflow_event`（payload 包括 actor、reason、refs）。

## 5. 哪些事件由 AI 触发

- ai_generate_started （由 AI 启动时记录 AiRecord.status -> running）
- ai_generate_completed （AiRecord.status -> completed; Service 处理该事件并将 Task 状态从 ai_working -> waiting_lawyer）
- ai_revision_started
- ai_revision_completed
- ai_error
- estimate_suggestion（AI 提供 estimated_minutes/ confidence，这为 Today 提供信号，但不写回 Task，除非 Service 层以 Timeline 事件写入）

AI 触发的事件应创建或更新 `AiRecord`，并通过 WorkflowEvent/Timeline 将摘要/summary/refs 写入，便于 Today 读取。

## 6. 非法状态流转规则

- 不允许 AI 直接将 `waiting_lawyer` -> `finalized` 或 `finalized` -> `completed`。`finalized` 与 `completed` 必须由律师触发（`lawyer_finalize`, `mark_completed`）。
- 不允许从 `completed` 回退到非终态，除非 `force_transition`（仅 admin）或产生新的 Task。
- 不允许跳过 `waiting_lawyer` 而直接由 AI 标记 `approved` 或 `finalized`。
- 不允许在 `waiting_materials` 时将 Task 移至 `finalized` / `approved`（材料缺失必须首先解决）。
- 对于 `error`：严禁忽略错误直接进入 `finalized`，必须先记录恢复事件与责任人。

Service 层应在 `updateByTaskId` 前进行校验：如果请求的 `status` 不在允许的目标集合里，返回 400 并写 `timeline` 记录拒绝原因。

## 7. 如何记录 history（先复用 ai_records / timelines / workflow_events，仅设计）

原则：所有 Command 与 Internal event 都必须写入 audit trail。优先使用已存在表，并在 Service 层统一封装写入逻辑。

- Write Timeline entries for every state change
  - Timeline fields to populate: { timeline_id, matter_id, task_id, event_type: 'task:status_change', event_time, description, source: actor_type, status: new_state }
  - Description 结构化约定：JSON string 包含 { from, to, actor: {id,type}, reason?, refs: {ai_record_id?, workflow_event_id?}, payload? }

- Write WorkflowEvent entries for higher-level commands that represent business processes
  - Use `workflow_event.event_type` values: 'task:start', 'task:ai_generate', 'task:lawyer_approve', 'task:request_revision', 'task:finalize', 'task:assign'
  - Store `payload` JSON with detailed meta (instructions, ai_summary, estimate)

- Use AiRecord to store AI-run artifacts
  - On AI start: create AiRecord with `ai_task_type`, `status=running`, `model`, `prompt_uri` if any
  - On AI completion: update AiRecord with `status=completed`, `result_uri`, and include `meta` in WorkflowEvent.timeline payload (e.g., `change_summary`, `revision_round`, `confidence`)

- Cross-linking
  - Timeline.workflow_events <-> WorkflowEvent.timeline_id should be set where appropriate
  - AiRecord.task_id should be set for traceability

- Read model
  - TodayService and Dispatch Engine should call a TaskService aggregation method (see V1 Implementation) that returns { task, last_timeline_events, latest_ai_record_summary, workflow_event_summaries }

Security / Data Hygiene
- Timeline description JSON must avoid embedding large AI outputs; store large artifacts URIs in AiRecord.result_uri.

## 8. 现有 Task 字段如何映射

- `task_id` -> Task.task_id
- `matter_id` -> Task.matter_id
- `title` -> 用作 UI 显示与短任务描述（可包含内部 type 前缀）
- `description` -> 人类可读说明；Service 层可以约定 JSON 前缀/后缀存结构化辅助信息，但不推荐长期使用
- `priority` -> 直接映射 priority 字段（约定值集合： low/normal/high/urgent）
- `due_date` -> 截止日期展示与 overdue 规则
- `status` -> 存放当前状态机的字符串（V1 约定使用 P2.2 列表中的值）
- `client_id`, `material_id`, `evidence_id`, `document_id` -> 关联对象引用
- `created_at`, `updated_at` -> 时间线参考

注：在 V1 中，我们约定把状态枚举字符串写入 `task.status`，并让 TaskService 在写入前进行校验与 Timeline 写记录。

## 9. 当前缺口

从 M160.1 的审计与本状态机设计可见的缺口：

- 缺少 `task_type`、`owner_user_id`、`estimated_minutes`、`revision_round`、`waitingReason` 等结构化字段，导致查询与规则复杂度高。
- `description` 被滥用作半结构化存储；需尽量改用 `timeline/workflow_event` 写入结构化 payload。
- Repository/Service 缺少聚合查询（Task + AiRecord + Timeline）返回 DTO 以便 Today 使用（需在 Service 层实现聚合方法）。
- 缺少对 `task.status` 写入前的状态验证逻辑（当前 TaskService 直接 updateByTaskId）。
- Frontend 未统一 Task UI，Today 依赖推断字段在短期内必须通过 Service 聚合提供。

## 10. 不改 Schema 的 V1 实现方式（步骤与约定）

概览：所有实现仅在 Service 层与 Workflow 层完成；不改 Prisma Schema、不新增表。

A. 在 `TaskService` 中新增/扩展方法（实现于应用层，不改 Repo Schema）
- `transitionTaskStatus(task_id, actor, event, payload)`：
  - 负责校验当前状态、根据 event 计算 newState（基于本设计的状态流表）、对非法流转返回错误并写 Timeline。
  - 写入 Task.status（prisma.update），并在成功后写 Timeline + WorkflowEvent（payload 包含 actor/refs/reason）。
  - 当 AI 相关事件发生，创建/更新 AiRecord 并把 ai_record_id 写入 WorkflowEvent/TImeline payload。

- `listByMatterWithMeta(matter_id)`：
  - 聚合查询 Task + 最近 N 条 Timeline + 最新 AiRecord（按 created_at desc）+ 最近 WorkflowEvents，用以构造 Today DTO。

B. 约定：所有 Task status 的写入入口只能通过 `TaskService.transitionTaskStatus`（或路由层校验后）
- 禁止直接调用 `TaskRepository.updateByTaskId` 改写 `status`（除非 admin 或 migration 工具）。

C. Timeline / WorkflowEvent 写入规范（Service 层封装）
- `TaskService` 在每次状态变更中同时写入 Timeline 与 WorkflowEvent：
  - Timeline.event_type = 'task:status_change'，description JSON 包含 { from, to, actor, reason, refs }
  - WorkflowEvent.event_type 对应业务命令，如 'task:ai_generate'、'task:request_revision' 等，payload 包含 ai_summary 或 instructions

D. AI Integrations
- AI 运行由 AI 模块创建 AiRecord（包含 task_id）；AI 完成后调用 `TaskService.transitionTaskStatus(task_id, system_ai, 'ai_generate_completed', { ai_record_id })`。
- AI 如果提供估时/confidence，Service 层将该信息写入 WorkflowEvent.payload（方便 Today 使用），而不直接改写 Task 表。

E. Frontend 与 Today
- 前端调用新的聚合接口 `listByMatterWithMeta`，而非直接读取 `taskRepository.findByMatterId`。
- TodayService 在后端实现规则，把 Task DTO 分成 completed/waiting/nextActions/activeMatters，并暴露 `GET /today/dashboard`（已在 M150.1 设计中）。

F. 权限与审计
- `transitionTaskStatus` 对律师动作强制 actor 验证与权限检查；对 AI 事件进行来源校验（确保 ai_record 属于本系统/模型）。

## 11. 将来需要 Schema 增强的候选字段

（为 M160.3 做准备）

优先级高（推荐在 V2 添加）
- `task_type` STRING/ENUM — 明确 Task 类型，便于索引与聚合
- `owner_user_id` STRING — 责任人
- `meta` JSON — 任意键值对（estimated_minutes, waitingReason, required_inputs）
- `revision_round` INT — 当前修订轮次
- `estimated_minutes` INT — 预计耗时
- `status` ENUM — 强化为受限枚举或使用 SystemEnum 进行映射

优先级中等
- `priority_score` FLOAT — 计算型优先级分数（由 Dispatch 写入）

可选（审计/运维）
- `last_ai_record_id` STRING — 最近一次 AiRecord 引用（便于快速联表）

## 12. M160.3 实施建议

目标：把本设计落地到代码（Service 层）与运维计划。

步骤（高层）：
1. 开发任务（短期）
  - 在 `apps/backend/src/services/taskService.ts` 增加：
    - `transitionTaskStatus(task_id, actor, event, payload)`
    - `listByMatterWithMeta(matter_id)`（Task DTO 聚合）
    - 内部封装 `writeTimeline` 与 `writeWorkflowEvent` 工具
  - 在 `apps/backend/src/repositories/taskRepository.ts` 保持 CRUD，但限制对 `status` 的直接写入（以 code review 与团队约定为强制）。

2. 集成测试
  - 编写端到端测试覆盖关键流转：AI 生成 -> 律师请求修订 -> AI 修订 -> 律师批准 -> 定稿 -> 完成。
  - 覆盖非法流转拒绝用例（AI 尝试 finalize、completed 回退等）。

3. Backfill / Migration 工具（非 Schema 改动）
  - 提供后台任务，用 Timeline / AiRecord 填充 `task.status` 为标准化枚举（针对历史数据）

4. Frontend 调整
  - 将前端 Task 列表/Workspace/Today 从直接读取 `findByMatterId` 切换为 `listByMatterWithMeta`。
  - UI 以友好短语展示状态并在 Task 操作时调用后端 transition API（或现有 Task endpoints 通过 Service 层改造）。

5. 监控与度量
  - 增加监控：状态转移延迟、AI error rate、平均 revision_round
  - Today Dashboard 显示关键 KPI，便于持续优化 Dispatch 规则

交付件（短期）
- 文档：此设计文档、API 规范草案（仅 Service contract）、端到端测试计划
- 代码：仅 Service 层实现（`TaskService.transitionTaskStatus`、 聚合查询方法）
- Tests：覆盖流转与非法流转

风险与缓解
- 风险：直接写入 `task.status` 可能与历史数据不一致；缓解：先实现 Service 层校验，逐步 backfill 历史 tasks 为标准 status。
- 风险：AI 写入 timeline 泄露大量文本；缓解：在 Timeline 只写摘要与 result_uri，AiRecord 存储完整结果。
