# P2.5 Task / Work Item Specification

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: 定义 LawDesk 中 `Task`（Work Item）在概念、生命周期、角色与展现层面的统一规范。设计-only：不改代码、不改 schema、不新增表、不实现 API 或 UI。

核心定义（简要）

- Today 调度 Matter。
- Matter 管案件与边界。
- Task 管具体工作（Work Item）。
- AI Work（AiRecord）执行或协助 Task。 

重要原则

- 律师界面不展示术语 `Task` / `Work Item`，显示为业务友好术语（例如：证据整理、法律检索、起诉状草稿、庭审准备、执行准备）。
- 技术层内部统一使用 `Task` 概念以保证可组合的规则引擎与 Today 聚合。
- 不再为更高层创造新的抽象，所有工作以 `Task` 为最小统一单元。

1. 为什么需要 Task / Work Item

- 统一可被调度与计量的“单元”：Task 提供 Today、Dispatch 与 Reporting 所需的最小可操作单元。
- 解耦展示与执行：律所产品可将用户友好语义映射到 Task 类型，而规则引擎以 Task 为统一输入。
- 支持 AI 与人协作：Task 明确责任、所需输入、期望产出与审计点，便于 AI 执行与审阅循环。

2. Task 与 Matter 的关系

- 每个 Task 必须归属于一个且仅一个 `Matter`（聚合根）。
- Matter 提供权限、审计上下文与生命周期边界；Task 不能脱离 Matter 存在。
- Matter 汇总其下 Task 的状态来计算活跃度、优先级与风险。

3. Task 与 Workspace 的关系

- Workspace 是 Matter 下按角色组织的视图：Evidence/Research/Documents/Execution 等。
- Task 指向一个或多个 Workspace（`targetWorkspace`），并在跳转时打开对应视图的条目或创建界面。
- Workspace 展现 Task 的友好标签与操作入口（Start / Review / Approve / Finalize）。

4. Task 与 Artifact 的关系

- Artifact 是 Task 完成后产生的可审计产物（文书草稿、证据条目、事实清单等）。
- 一个 Task 可产生 0..N 个 Artifact；Artifact 反向引用其源 Task（若有）。
- Artifact 的版本控制与 `finalized` 状态由律师驱动，Artifact 成为法律事实或可提交材料时必须有审计快照。

5. Task 与 AI Work 的关系

- AI Work（AiRecord）是 AI 针对某个 Task 的执行记录与中间产物（生成草稿、分析报告、自检结果）。
- AI Work 与 Task 的关联必须可追溯：包含 `engine_version`, `revision_round`, `confidence_score`, `meta`。
- AI Work 可触发 Task 状态流转（例如：`ready_to_start` -> `ai_working` -> `waiting_lawyer`），但不得直接 `finalize` 或 `complete`。

6. Task 类型（示例/非穷举）

- evidence整理（evidence collection / structuring）
- facts归纳（fact extraction / summary）
- legal research（法条/判例检索）
- complaint draft（起诉状草稿 / pleading）
- argument draft（论证/代理词草稿）
- hearing preparation（庭审准备）
- execution preparation（执行准备）

说明：前端显示用语应为律师熟悉的业务名称（中文短语），类型仅作为规则与映射的内部标签。

7. Task 生命周期（统一状态枚举）

- `waiting_materials` — 等待外部或客户补充关键材料
- `ready_to_start` — 材料齐备，律师或 AI 可开始
- `ai_working` — AI 正在处理/生成中
- `waiting_lawyer` — 等待律师复核/补充/判断
- `revision_requested` — 律师请求修改
- `ai_revising` — AI 根据指令进行修订
- `approved` — 律师审核通过（非定稿）
- `finalized` — 律师定稿（不可被 AI 自动覆盖）
- `completed` — 最终完成（如已提交/执行)

注：UI 映射应将这些内部枚举转为友好文本（示例见 P2.2）。

8. 律师动作（映射到生命周期）

- Upload Materials：`waiting_materials` -> `ready_to_start`（当材料满足）
- Start（同意开始）：`ready_to_start` -> `ai_working` 或人工处理
- Review / Approve：`waiting_lawyer` -> `approved`
- Request Revision：`waiting_lawyer` -> `revision_requested`
- Add Materials：任意 `waiting_*` -> `ready_to_start`
- Finalize：`approved` -> `finalized`
- Complete（如提交/送达）：`finalized` -> `completed`

所有律师动作须产生审计记录（who/when/why，及引用的 ai_work metadata）。

9. AI 动作（映射到生命周期）

- Process（数据清洗/抽取）：触发 `ai_working`
- Generate（草稿/摘要）：`ai_working` -> `waiting_lawyer`（附带 `change_summary`）
- Self-check（自检）：产生 `self_check_report`，随 ai_work 一并记录
- Revise：响应 `revision_requested` -> `ai_revising` -> `waiting_lawyer`
- Estimate（预计耗时/confidence）：作为 meta 提示给 Today 的优先规则

约束：AI 不得执行 `finalize` 或 `complete`。

10. Today 如何读取 Task

- TodayService/DispatchEngine 以 Task 为首要信号源之一，按规则将 Task 分入：`waiting` / `nextActions` / `completed` / `activeMatters`。
- 重要字段：`task_id`, `matter_id`, `type`, `status`, `priority`, `updated_at`, `revision_round`, `waitingReason`, `estimatedMinutes`。
- Today 不写回 Task；它仅基于 Task 与其关联对象（Artifact/AiRecord）计算优先级与建议。

11. Matter 首页如何读取 Task

- Matter Overview 汇总该 Matter 下的 Top Waiting Tasks、Top Next Actions 与近期 Completed Tasks。
- 显示聚合指标：waitingCount、overdueCount、avgRevisionRounds（样例指标）以便律师快速判断总体进度。

12. Workspace 如何读取 Task

- Workspace（如 Documents Tab）按 Task.type 或 targetWorkspace 过滤显示相关 Task 列表（例如只看 documents 相关的 tasks）。
- 条目详情展示：status（友好文本）、revision_round、last_updated、waitingReason、linkedArtifacts、ai_work_metadata 与快速操作按钮（Start/Request Revision/Approve/Finalize）。

13. V1 实现边界

- V1 只定义 Task 规范与 Today/Dispatch 的只读聚合；不新增持久化表或变更现有 schema。
- V1 不实现自动化批量审批或 AI 批量执行的写回逻辑。
- V1 保持可解释的规则引擎（rule_v1），不接入 LLM 作为优先级黑箱。

14. V2 扩展方向

- 可选：将 Task 的部分规则可配置（权重可调），在严格审计与权限下支持授权式半自动执行。
- 可选：引入 Task 模板与 SLA（超时提醒、分配策略、负责人继承）。
- 可选：引入更细粒度的质量评分（基于历史 AI 修订成功率）与 ML 优化 priorityScore。

15. 验收标准

- 文档明确 Task 概念、与 Matter/Workspace/Artifact/AiRecord 的关系（第1–6节）。
- 列出 Task 类型并说明前端应使用的友好显示语（第6节与重要原则）。
- 提供统一生命周期枚举与状态说明（第7节）。
- 明确律师/AI 的动作与权限映射，并要求审计（第8、9节）。
- 说明 Today/Matter/Workspace 以 Task 为信号的读取方式（第10–12节）。
- 明确 V1 边界与 V2 扩展方向（第13、14节）。
- 包含 3 个可执行验收场景示例：
  - 场景 A（空文书草稿）：Task(status=draft, content empty) -> Today shows waitingReason -> lawyer edit -> finalize -> completed
  - 场景 B（AI 多轮修订）：lawyer request revision -> ai_revising x N -> approve -> finalized; revision_round incremented
  - 场景 C（逾期 Task 优先）：overdue task appears in Top Action -> lawyer resolves -> completed recorded
