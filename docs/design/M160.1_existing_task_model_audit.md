# M160.1 Existing Task Model Audit

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: 审计现有 `Task` Prisma 模型及相关后端/前端使用情况，评估其能否承载 P2.5 中定义的 Task / Work Item 机制（只做审计、无代码或 schema 改动）。

Summary (结论摘要)

现有 `Task` 模型在基础 CRUD 与与 Matter 关联方面完备，可作为 P2.5 的最小承载单元；但模型缺少若干用于 Today/Dispatch 与 AI 协作的结构化元数据字段（例如 `task_type`, `waitingReason`, `revision_round`, `estimatedMinutes`, `owner_user_id`, `meta` JSON 等），且前端并未采用统一 Task UI（使用友好视图与 Next Actions 代替）。

短期（V1）结论：可通过约定与利用现有关联表（`AiRecord`、`Timeline`、`WorkflowEvent`、`Workspace`）以及统一的 status 值策略，在不改 Schema 的前提下实现 P2.5 的大部分可读/调度功能；但会牺牲结构化查询效率与部分可解释性。

长期（V2）建议：为 Task 增加若干字段（或一个 `meta` JSON 字段）以保存 revision_round、waitingReason、estimatedMinutes、task_type、owner 等；或增设 TaskMetadata 表以提高查询性能与可维护性。

## 1. 当前 Task 数据结构（来自 `packages/database/prisma/schema.prisma`）

Task 模型字段：
- `id` (uuid) — PK
- `task_id` (string, unique) — 业务 ID
- `matter_id` (string) — 关联 Matter
- `client_id?`, `material_id?`, `evidence_id?`, `document_id?` — 可选外键引用
- `title` (string)
- `description` (string)
- `priority` (string)
- `due_date?` (DateTime)
- `status` (string)
- `created_at`, `updated_at`

关联（关系）：
- relation -> `Matter`, `Client?`, `Material?`, `Evidence?`, `Document?`
- 反向关系：`Timeline`, `WorkflowEvent`, `AiRecord`, `Knowledge`, `Workspace`

备注：没有 `type`、没有 `owner`、没有 `meta`（JSON）字段、没有专门的 `revision_round` 或 AI-metadata 字段。

## 2. 当前能力（Repository / Service / Routes）

- `apps/backend/src/repositories/taskRepository.ts`：
  - 支持 `findByMatterId(matter_id)`、`create(data)`、`updateByTaskId(task_id, patch)`、`deleteByTaskId(task_id)`。
  - `create` 接受常见字段并设置默认 `priority='normal'`、`status='open'`。

- `apps/backend/src/services/taskService.ts`：
  - 封装 repository；提供 `listByMatter`、`createForMatter`、`updateByTaskId`、`deleteByTaskId`。
  - 无额外业务逻辑（纯 CRUD 封装）。

- `apps/backend/src/routes/matterRoutes.ts`：
  - 暴露 REST：`GET /matters/:id/tasks`、`POST /matters/:id/tasks`、`PATCH /matters/:id/tasks/:task_id`、`DELETE /matters/:id/tasks/:task_id`。
  - 没有复杂的查询过滤或 Task-specific rules。

- Frontend：
  - `apps/frontend/src/app/matters/[matter_id]/page.tsx` 存在 `TaskCard`/`RichTaskCard` 组件逻辑与“下一步建议”视觉，但并未以统一 Task 管理界面暴露所有 Task 字段；页面更多基于 Documents/Evidence/Facts 等资源构建建议与操作入口。
  - 其他 Workspace 页（`facts`, `execution`, `documents`, `evidence` 等）在 UI 中以各自资源为主；Task 被间接利用（Next Actions、Runtime/Execution 行为）而非直接作为首要展示对象。

## 3. 可复用部分（现有可以直接利用以满足 P2.5 的点）

- 关系完整性：Task 已与 `Matter`、`Document`、`Evidence` 等对象建立外键，可直接作为 Work Item 与这些 Artifact 互相引用。
- AiRecord 关系：`AiRecord` 模型支持 `task_id`，可用作 AI 执行记录的溯源（适合作为 AI Work 的存放处）。
- Timeline / WorkflowEvent：可记录审计动作与任务状态变化（用于审计与历史回溯）。
- 基础 CRUD API：已有创建/更新/查询/删除接口，前端可调用以实现 Task 创建与变更。
- 优先级字段：`priority` 字段可直接映射为 P1/P2/P3 等（需约定值集）。

## 4. 不足（Gap Analysis）

1. 缺少 `task_type` 字段：现有模型没有明确类型字段（evidence整理、facts 归纳、legal research 等），只有 title/description。无法高效按类型聚合或筛选。
2. 缺少结构化 AI/Workflow 元数据：没有 `revision_round`, `waitingReason`, `estimatedMinutes`, `requiredInputs`, `confidence_score` 等字段。
3. 无 `owner_user_id` / assignee 字段：无法直接在 Task 层面绑定责任律师/处理人（当前可用 Timeline 或 Client 绕行但不直观）。
4. `description` 是纯文本：若要承载结构化 `requiredInputs` 或 `waitingReason`，会被迫序列化 JSON 到字符串，影响查询与可读性。
5. `status` 为自由字符串：需要统一枚举以支持 P2.2 中的生命周期（waiting_materials, ready_to_start, ai_working, ...），当前并未强制/列出枚举。
6. Repository/Service 层缺少基于统一生命周期与 Task-type 的查询/过滤能力（例如按 `task_type`、按 `revision_round`、按 `waitingReason` 过滤）——需在应用层实现约定或扩展查询接口。
7. 前端缺少真正以 Task 为中心的 UI：当前以 Workspace 视图与 Next Actions 为主，Task 的存在感较低，且 UI 并未展示 ai metadata 或 revision round 等信息。
8. 可审计/可解释性字段分散：AI 元数据常驻于 `AiRecord`，但没有强制把 summary/summary_score 等关键字段同步到 Task 以便 Today/Dispatch 更快地读取。

## 5. 是否能直接承载 Work Item（能 / 不能 的说明）

- 能（在 V1 范围内）：
  - Task 与 Matter 的外键、AiRecord/TImeline 关联，及现有 CRUD 接口，足以承载 P2.5 的核心概念（每个 Work Item 属于一个 Matter，AI Work 通过 AiRecord 关联 Task）。
  - 可通过约定将 `status` 值映射为统一生命周期（P2.2 的枚举），并在 Dispatch Engine / TodayService 中以此进行规则计算。

- 但不够理想（限制）：
  - 缺少结构化元数据导致 Today/Dispatch 读取与排序需要更多联表查询与非结构化解析（例如解析 description 中的 JSON），降低性能并增加实现复杂度。
  - 缺少 owner 与估时字段导致无法在 Today 中直接给出责任人与预计耗时提示，影响首屏“今天我该做什么”的体验。

## 6. 是否需要新增字段或新增表（建议）

短期（V1，实现无需改 Schema）：
- 无需新增表或字段。通过以下约定可实现 P2.5 的大部分能力：
  - 统一 `status` 值集（由应用层约束）：采用 P2.2 的生命周期字符串作为 `status` 值（`waiting_materials`、`ready_to_start`、`ai_working`、`waiting_lawyer`、`revision_requested`、`ai_revising`、`approved`、`finalized`、`completed`）。
  - 使用 `AiRecord` 存放 AI-specific metadata（engine_version、revision_round、confidence、change_summary、self_check_report）并在 TaskService 聚合查询时联表返回。
  - 使用 `WorkflowEvent` / `Timeline` 记录 assignment、estimatedMinutes、waitingReason 等可审计事件（作为替代 owner/metadata 存储）。
  - 定义在 Task.title/description 与 Timeline 中的约定（例如 JSON schema 约定）以临时记录额外属性（注意：性能/可读性折中）。

长期（V2，推荐 Schema 改造）：
- 新增字段（建议）：
  - `type` 或 `task_type` (string / enum)
  - `owner_user_id` (string) — 责任人
  - `meta` (Json) — 任意键值用于 `requiredInputs`, `estimatedMinutes`, `waitingReason`, `etaMinutes` 等
  - `revision_round` (int)
  - `estimated_minutes` (int)
  - `status` 强化为枚举（或用 `system_enums` 做映射）
- 或者新增表：`task_metadata`（task_id -> key:value rows）以支持灵活查询与索引

理由：这些字段能显著简化 Today/Dispatch 的读取逻辑并提高查询性能与可解释性。

## 7. 推荐 V1 改造方案（不改 Schema，仅应用层约定）

目标：在不改 Schema 的前提下，快速让现有系统符合 P2.5 规范并在 Today 中高效工作。

A. 约定与规范（立即实施，无 DB 改动）
- 统一 `status` 值：所有写 Task 的代码路径必须使用 P2.2 的生命周期枚举字符串作为 `status`（在服务层做映射校验）。
- 约定 `task_type`：使用 `title` 或 `description` 前缀/标记约定（短期）并在 Service 层提供 `inferTaskType(task)` 方法以供 Dispatch 使用。
- AI 元数据存放：所有 AI 输出必须创建对应的 `AiRecord` 并保证 `task_id` 关联；AiRecord 包含 `revision_round`, `confidence_score`, `engine_version`, `change_summary`。
- 使用 `WorkflowEvent` / `Timeline` 记录 assignment、estimatedMinutes、waitingReason。约定事件 `workflow_event.event_type` 值集（如 `task:assign`, `task:waiting_reason`, `task:estimate`）。

B. Backend Service 层增强（仅应用层改动）
- TaskService 增加 `listByMatterWithAiMeta(matter_id)` 聚合方法，联表查询 AiRecord / Timeline / WorkflowEvent，返回统一 DTO：{ task_id, title, status, task_type, priority, revision_round, waitingReason, estimatedMinutes, owner, ai_meta }
- Dispatch Engine / TodayService 使用该 DTO 以避免前端或多次后端调用。

C. Frontend 约定与显示
- 前端不直接解析 `description` 的结构化内容，改为调用新的聚合 DTO 接口（由 TaskService 提供）以获得标准字段。
- 在短期内，仍维持用户可读的业务术语（证据整理等），UI 层把 `task_type` 内部标签映射为友好字符串。

D. 数据迁移策略（若有）
- V1 无需迁移；若有历史任务需要补充 metadata，可通过后台脚本把常见信息写入 Timeline/AiRecord，以后逐步迁移到结构化字段（V2）。

## 8. M160.2 建议（后续行动项）

- M160.2（建议实施）：设计并评估 V2 Schema 改造方案，具体包括：
  1. 为 `Task` 增加 `task_type`, `owner_user_id`, `meta`(Json), `revision_round`, `estimated_minutes` 字段，或创建 `task_metadata` 表。
  2. 评估索引与查询成本，定义迁移脚本以把 Timeline/AiRecord 中的元数据写入新字段（可选分阶段迁移）。
  3. 更新 Repository/Service/Routes：提供聚合查询接口（Task + AiRecord + Timeline）并在 TaskService 中暴露 DTO。
  4. 前端：设计以 Task 为第一等公民的 UI（Task 列表、Task 详情、task 操作面板），并在 Today 中显示 `estimatedMinutes`、`owner`、`waitingReason`。

- 为 M160.2 编写小范围影响评估（列表受影响的 API、前端路由、现有脚本与运维），并评估是否需要迁移窗口与兼容层（feature flags）。
