# M161.2 Matter Progression Rules (V1)

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: 定义 Matter Engine 在 V1 场景下的推进/暂停/回退/完成规则（设计-only）。遵循约束：不改代码、不改 Schema、不新增表、不实现 API/UI。

摘要

本文件在 M161.1、P2.2 与 P2.5 基础上，明确 Matter 在案件生命周期中何时推进 Stage、何时暂停、何时回退与何时完成。目标是为 MatterEngine 提供可实现的规则集合（V1 以 Service 层实现，使用 `tasks`/`workflow_events`/`timelines`/`ai_records` 作为事实来源）。

约束与原则

- Matter Stage 不因单个 Task 的小修改而轻易回退；仅在关键 Task 状态变化或外部事件时才变更 Stage。 
- Task 支持多轮修改（revision_round），Task 内部可来回变更，但 Stage 推进以关键 Task 的完成为条件。
- 律师审核或需补资料会暂停 Matter 推进（写入 Timeline 标记等待原因）。
- Today 与 Dispatch 为只读消费端，不在此设计中写回状态。

## 1. Matter Stage 列表

## 3. Stage 推进规则

总体原则：仅在当前 Stage 的“关键 Task”被完成（`finalized` 或 `completed`）且无待办高优先事项时，Matter 才推进到下一 Stage。

规则细则（V1）：

- 定义关键 Task 集：每个 Stage 由一组 1..N 个关键 Task 组成（按上表），MatterEngine 使用该组判断推进条件。
- 推进条件（必须同时满足）：
  1. 所有关键 Task 的状态均为 `finalized` 或 `completed`；
  2. 该 Matter 下无 `waiting_lawyer` / `waiting_materials` 状态的高优先 Task（优先级 `urgent`/`high`）；
  3. 无开放的阻塞 WorkflowEvent（如 `court_hold`、`external_dependency`）；
  4. 未触发人工暂停（管理员/律师明确标记暂停）。
- 推进动作：
  - 写入 `Timeline`（event_type=`matter:stage_change`，payload 包含 from/to/refs/actor/reason）
  - 可选：更新 `Matter.status` 为新的 Stage 字符串（建议写，但不是必须）
  - 创建下一 Stage 的初始默认 Task（见第2节）— 仅在策略需要时创建（可为 lazy create）。

幂等性：基于 `workflow_event_id` 或 `stage_change_idempotency_key` 防止重复推进。


## 4. Stage 暂停规则

何为暂停：暂停使 Matter 不再自动推进至下一 Stage，仍允许 Task 内部流转与 AI 多轮修订。

触发暂停的条件：

- 律师显式标记为暂停（manual_pause WorkflowEvent）；
- 出现外部阻塞（例如法院保全、第三方证据缺失，WorkflowEvent=`external_dependency`）；
- 存在关键 Task 处于 `waiting_materials` 且指定为阻塞（blocking flag），且未在合理时限内补齐；
- 发生严重 `error`（AI error 导致不可继续），需要人工干预。

暂停行为：

- MatterEngine 写入 `Timeline`（event_type=`matter:paused`，payload 包含 reason 与 refs）；
- 停止 Stage 自动推进逻辑；
- 可触发通知（V1 建议写 Timeline 并让上层系统读取并通知，而不直接推送）。


## 5. Task 修改时 Matter 是否回退（回退策略）

核心原则：避免因单个 Task 的小改动导致 Stage 回退。回退应仅在以下情形发生：

- 关键性回退条件（需要回退 Stage）：
  1. 关键 Task 被重置为 `waiting_materials` 或低于 `ready_to_start`，且其为 Stage 进行的前置条件，且不能通过并行 Task 补位；
  2. 发现外部事实或裁定使得已完成的 Stage 需重新处理（例如法院裁定需补充证据）；
  3. 人工强制回退（manual_force_revert，由有权限的律师/管理员触发）。

- 非回退条件（不触发 Stage 回退）：
  - 单个 Task 的 minor edit（内容修正、格式或 metadata 变更）；
  - Task 在 `waiting_lawyer` 与 `revision_requested` 之间多轮往返；
  - 非关键 Task 的状态变动（非关键 Task 可在不影响 Stage 的情况下被重新打开或修改）。

回退流程（若满足回退条件）：

1. MatterEngine 在写入回退 Timeline（event_type=`matter:stage_revert`，包含 from/to/actor/reason）后，将下一步由系统或律师决定是否重建 Stage 的关键 Task（可能需要创建替代 Tasks）；
2. 回退操作应记录强制来源与理由（审计）；
3. 回退后应触发一次健康检查（Health WorkflowEvent）以计算是否需要人工复核。 


## 6. 等待律师规则

何时标记等待律师：

- Task 进入 `waiting_lawyer`（AI 生成草稿或人工操作后提交审核）；
- 律师请求审阅/批准时标记 `waiting_lawyer`；
- 任何 `waiting_lawyer` 与 `revision_requested` 的组合表示 Matter 不应自动推进 Stage。 

影响与行为：

- MatterEngine 将避免推进 Stage，直到相关关键 Task 从 `waiting_lawyer` 转为 `approved`/`finalized` 或经过人工 override；
- 写入 Timeline（event_type=`matter:waiting_lawyer`，payload 含 task_ids、waitingReason 与 actor）；
- Today 将把这些 Task 列为 `waiting` 分组，供律师处理。 

超时策略（V1 推荐）：

- 可在 MatterEngine 中记录 `waiting_lawyer` 的开始时间，若超过阈值（可配置，如 72 小时）生成 `escalation` WorkflowEvent 提示负责人但不自动变更 Stage。


## 7. 等待补资料规则

当 Task 状态为 `waiting_materials`：

- MatterEngine 判断是否为阻塞：若 Task 被标记 `blocking=true` 且属于关键 Task，则 Matter 进入 `waiting` 状态并暂停推进；
- 写入 Timeline（event_type=`matter:waiting_materials`，payload 包含 missing_fields 或 required_materials 列表）；
- 重试与超时：若在阈值内补齐（由 `client_upload`/`lawyer_upload` 触发），MatterEngine 将调用 TaskService.transitionTaskStatus 至 `ready_to_start` 并继续评估推进规则；否则触发 `escalation`。 

数据要求（V1）：

- 在 Timeline.payload 中结构化写入缺失项列表（避免把大文本写入 Timeline，使用 AiRecord.result_uri 存大 artefact）。


## 8. completed / closed 规则

Stage completed 条件：

- Stage 下所有关键 Task 均为 `finalized` 或 `completed` 且无阻塞或 `waiting` 高优先事项。

Matter closed 条件：

- 所有 Stage（到 `closing`）完成并通过 `final_audit` Task（`closing:final_audit`）验证；
- 无未解决的 `external_dependency` 或 `appeal` 等待项；
- 由律师/管理员触发 `matter_closed` WorkflowEvent（明确该操作人）。

动作与审计：

- 写入 Timeline（event_type=`matter:completed` / `matter:closed`，payload 含 summary、refs）；
- 可选：更新 `Matter.status` 到 `closed` 并记录 `closed_at`（若项目政策允许写回）。


## 9. Matter Event 与 Task Event 的关系

（映射表 - V1 参考实现）

- `client_upload` / `lawyer_upload`  → 找到相关 Task (by material_id/evidence_id/document_id) → TaskEvent: `MATERIAL_ADDED` 或 `UPLOAD_COMPLETED`（若满足材料齐备则进 `ready_to_start`）
- `ai_draft_completed` → 相关 TaskEvent: `AI_FINISHED`（Task.status=`waiting_lawyer`）
- `lawyer_request_revision` → 相关 TaskEvent: `LAWYER_REVISION`（Task.status=`revision_requested`）
- `lawyer_approve` → TaskEvent: `LAWYER_APPROVED`（可能直接进 `approved`）
- `lawyer_finalize` → TaskEvent: `FINALIZED`（该 Task 进入不可自动覆盖终态）
- `deadline_reached` → 若 Task.overdue → create `escalation` Task 或 emit `START`（策略可选）
- `court_notice` / `external_dependency` → 可能触发 Matter-level pause（`matter:paused`）与创建关联 Task（如 compliance check）

实现注意：MatterEngine 应先尝试匹配现有 Task（by linked ids），找不到时再创建新 Task，并在每次映射/推进时写入 `WorkflowEvent` 与 `Timeline`。 


## 10. V1 不做什么

- 不自动将 `approved` 升级为 `finalized`（律师必须手动 finalize）；
- 不在数据库层新增 Stage 字段或新表；所有规则与判断在 Service 层实现并通过 Timeline 审计；
- 不实现实时 push 通知（仅写 Timeline / WorkflowEvent 供上层系统读取）；
- 不允许 AI 直接更改 Matter.stage 或直接 finalize/close Matter。


## 11. M162 代码实现建议（高层）

目标：为 M162（实现）提供明确、高内聚、可测试的实现路线，仍然不改 Schema。

建议实现组件：

- `apps/backend/src/services/matterEngine.ts`：核心规则引擎，导出 `handleWorkflowEvent(event)`、`evaluateMatter(matter_id)`、`attemptAdvanceStage(matter_id)`、`recheckBlockingTasks(matter_id)` 等函数。
- `MatterEngine` 实现要点：
  - 读取：使用 Repository 查询 `tasks`（按 matter_id）、`workflow_events` 与 `timelines`；
  - 判定：基于规则表判断推进/暂停/回退；
  - 写入：在决定后写 `Timeline` 与必要的 `WorkflowEvent`，并通过 `TaskService` 创建/推进 Task；
  - 幂等：使用 `workflow_event_id` 或 `event.idempotency_key` 记录已处理事件；
  - 错误处理：失败时写 `Timeline(error)` 并返回可重试的错误码。
- 调度：在 V1 使用轮询/cron（例如每 30s 或 1min）处理未消费的 `workflow_events`。避免在 V1 添加新 HTTP endpoints，除非需要人工回放/调试。 

测试建议：

- 单元：覆盖 `attemptAdvanceStage` 在不同 Task 集合与阻塞状态下的决策逻辑；
- 集成：注入 WorkflowEvent -> 运行 matterEngine.handleWorkflowEvent -> 断言 TaskService 被调用与 Timeline 写入；
- 回放测试：提供脚本重放历史 `workflow_events` 并断言幂等性与最终 Stage 状态。 


## 12. 验收标准

以下为 M161.2 的验收条件（必须全部满足）：

1. 文档包含所要求的 Stage 列表与每个 Stage 的默认 Task（第1、2 节）；
2. 明确且可操作的 Stage 推进规则、暂停规则与回退策略（第3、4、5 节）；
3. 明确等待律师与等待补资料的行为、超时与 escalation 处理（第6、7 节）；
4. 明确 completed / closed 的触发条件与审计动作（第8 节）；
5. 提供 Matter Event ↔ Task Event 的映射参考表（第9 节）；
6. 列出 V1 明确不做的项（第10 节）；
7. 给出 M162 的高层实现建议（第11 节）以便工程落地；
8. 所有规则均以 Timeline/WorkflowEvent 写审计，不依赖 Schema 变更；
9. 包含 3 个可执行验收用例（场景），并说明预期 Timeline 记录点与 Task 状态变化：
   - 场景 A（证据补齐推进）：client 上传材料 -> `client_upload` -> 更新关联 Task -> Task `MATERIAL_ADDED` -> Task `ready_to_start` -> 若关键 Task 全部 `finalized`，Stage 推进。
   - 场景 B（AI 修订循环）：触发 AI -> Task `ai_working` -> `ai_draft_completed` -> Task `waiting_lawyer` -> 律师 request revision -> `revision_requested` -> AI revise -> `ai_revising` -> `ai_draft_completed` -> 律师 approve -> `finalized`。
   - 场景 C（等待律师超时）：Task `waiting_lawyer` 超过阈值 -> MatterEngine 写 `escalation` WorkflowEvent，并在 Timeline 添加 `escalation` 条目（不自动推进）。
