# P2.3 AI × Lawyer Collaboration

- Date: 2026-07-10
- Author: LawDesk Team
- Scope: Define collaboration boundaries, action responsibilities, review loops and finalization mechanics between AI and lawyers in LawDesk. Design-only: no code, schema, table, API or UI changes.

References:
- P2.1 LawDesk Core Architecture
- P2.2 Unified Lifecycle

## 1. 核心原则：律师驾驶，AI 操作

- 律师负责方向、判断、批准与定稿（`lawyer-in-the-loop`）。
- AI 承担重复性加工工作：整理、分析、起草、修订与提交给律师审核。
- 任何不可逆或具有法律后果的操作必须由律师显式执行。AI 仅能建议与草稿。

## 2. 律师职责

- 决定何时开始处理（同意开始）。
- 上传并核对关键材料（保证证据链完整性）。
- 审核 AI 生成的草稿或建议（逐条判断法律要点、事实准确性）。
- 发出修订指令或补充资料请求。
- 执行 `Finalize`（定稿）与对外提交动作（提交法院、签署、导出）。
- 维护合规审计记录（who/when/why）。

## 3. AI 职责

- 对已有材料做质量检查与结构化（加工）。
- 从材料中抽取事实、梳理证据关联、生成初步事实清单（分析）。
- 起草文书草稿、论证要点、证据摘要（生成）。
- 内部自检（syntax/consistency/basic-legal-heuristics）并返回变更摘要（自检）。
- 根据律师指令进行修订与再修正（修正/再修正）。
- 按约定模板或指令提交成果供律师审核（提交审核）。

## 4. 律师动作（映射）

- 上传资料（Upload Materials）
  - 目的：补全输入数据，解除 `waiting_materials` 阻塞。
  - 审计：记录上传者与时间。

- 同意开始（Start）
  - 目的：律师确认材料足够并触发 AI 处理（或手动开始工作）。
  - 状态：`ready_to_start` -> `ai_working`。

- 审核（Approve）
  - 目的：律师确认草稿在实质与事实层面可接受（并非定稿）。
  - 状态：`waiting_lawyer` -> `approved`。

- 要求修改（Request Revision）
  - 目的：指出问题并定义修订要求（可结构化或文本指令）。
  - 状态：`waiting_lawyer` -> `revision_requested`。

- 补充资料（Add Materials）
  - 目的：解决 AI 无法推进或需更多证据的情形。
  - 状态：`waiting_*` -> `ready_to_start`（或触发新的 AI 轮次）。

- 审核通过（Approve）
  - 目的：进入定稿前的确认节点，允许后续定稿。

- 定稿（Finalize）
  - 目的：律师执行的不可逆动作，生成 `final` Artifact、触发签署或提交。
  - 状态：`approved` -> `finalized`。

所有律师动作需在 UI 中以明确按钮/面板呈现并记录审计信息。

## 5. AI 动作（映射）

- 加工（Process）
  - 目的：清洗、抽取、结构化输入材料，为后续任务准备数据。
  - 触发点：材料上传后或 `Start`。

- 分析（Analyze）
  - 目的：识别缺口、提出 `waitingReason`（如空草稿、弱证据）。
  - 输出：quality hints、evidence gaps。

- 生成（Generate）
  - 目的：按模板/指令生成草稿或摘要（Document/Evidence/Research）。
  - 输出：草稿 Artifact（Draft），同时生成 `change_summary`。

- 自检（Review）
  - 目的：在提交律师前运行静态校验（语法、引用完整性、明显矛盾检测）。
  - 输出：`self_check_report`，作为律师审阅辅助。

- 修正 / 再修正（Revise）
  - 目的：根据律师 `revision_instructions` 进行 N 轮修订。
  - 输出：新草稿、`diff` 与 `change_summary`。

- 提交审核（Submit for Review）
  - 目的：完成一轮 AI 工作并将结果呈现给律师（进入 `waiting_lawyer`）。

AI 的每次关键输出应附带元数据：`engine_version`, `revision_round`, `self_check_report`, `confidence_score`（解释性字段）。

## 6. 审核循环

- 一轮审核（Single Review Round）
  1. 律师触发或同意 AI 生成（`Start`）。
  2. AI 生成草稿并自检，`Submit for Review` -> `waiting_lawyer`。
  3. 律师审核并选择 `Approve` 或 `Request Revision`。

- 多轮审核（Multi-round）
  - 若 `Request Revision`：律师提供 `revision_instructions`（可为结构化选项或文本），系统记录 `revision_round++`。
  - AI 进入 `ai_revising`，生成新草稿并自检，重复提交。
  - 限制策略：建议默认 5 轮软上限，超过上限提示人工介入并给出负责人建议。

- 补充资料后的再加工
  - 若律师在审核中 `Add Materials`：材料补充后 Work Item 可以回到 `ready_to_start` 并触发新一轮 `ai_working`（或手动 Start）。

## 7. 审核通过与定稿的区别

- 审核通过（Approve）
  - 含义：律师对草稿内容认可，但仍保留改进空间；不产生 `final` Artifact。
  - 结果：Work Item 状态转为 `approved`，可进入定稿流程。

- 定稿（Finalize）
  - 含义：律师执行终结性动作，生成 `final` Artifact，触发签署/导出/提交等不可逆流程。
  - 要求：必须记录版本快照与审计日志。

## 8. AI 不得做的事

- 任何具有法律后果的不可逆操作（提交法院、签署、删除证据等）。
- 在没有律师明确授权下，替代律师作出最终法律判断或发布法律意见。
- 修改或覆盖 `finalized` Artifact（除非律师手动撤回并重做流程）。
- 在未经授权的情况下将敏感数据外发或调用未批准的外部服务。

## 9. 律师必须确认的节点

- 从 `waiting_lawyer` 到 `approved`（审核通过）
- 从 `approved` 到 `finalized`（定稿/提交/签署）
- 任何删除、移除证据或关闭 Matter 的操作
- 批量执行 AI 建议（若实现半自动化）— 律师需单独授权

## 10. Workspace 中如何呈现协作状态

- 每个条目展示：友好状态（映射自统一状态枚举）、`revision_round`、`last_updated`、`waitingReason`、AI `confidence` 摘要。
- 在条目详情：展示 AI `self_check_report`、`change_summary`、修订历史与每轮审计记录。
- 操作面板：提供明确按钮：`Start`、`Request Revision`、`Approve`、`Finalize`、`Add Materials`。
- 对于律师：提供“快速合并/接受建议”短语与逐条接受选项，避免一次性盲批。

## 11. Today Dashboard 中如何呈现协作状态

- `waiting` 分组将包含需律师操作的条目（`waiting_lawyer`、`revision_requested`、`waiting_materials`）。
- `nextActions` 可包含建议：`开始 AI 生成`、`上传证据`、`处理逾期任务`。
- 每条 `waiting` 项显示简短理由（如：`待律师补全文书内容`）与可直接跳转的动作（进入对应 Workspace / 条目）。
- 对于高优先级/高风险项，显示 `riskHint`（例如 `逾期`、`空草稿`）。

## 12. Matter 首页中如何呈现协作状态

- 显示 Matter 层级摘要：当前 `waiting` 数量、当前 `revision_round` 平均值、是否有 `finalize` 待执行。
- 提供快速入口：进入最近一个 `waiting_lawyer` 条目或本日 `nextActions` 列表。

## 13. Documents / Evidence / Research / Execution 的协作示例

- Documents
  - 场景：AI 生成起诉状草稿 -> `ai_working` -> `waiting_lawyer`。
  - 律师查看 `change_summary`，逐段接受/要求修改 -> 多轮修订 -> `approved` -> `finalized` -> 提交法院。

- Evidence
  - 场景：客户上传银行流水（Upload Materials） -> 证据条目 `ready_to_start` -> AI 提取关键交易并关联事实 -> 律师确认关联（Approve）或补充说明（Add Materials）。

- Research
  - 场景：AI 聚合相关判例与法条（Generate），生成 `self_check_report` 与置信度，律师审阅并标注关键引用（Approve / Request Revision）。

- Execution
  - 场景：Execution 操作（如送达）仍由律师或系统管理员执行；AI 可以生成送达文本、清单与流程建议，但不得替代提交行为。

## 14. V1 实现边界

- V1 实现：
  - UI 映射友好状态并显示 AI 元数据（`confidence`, `revision_round`, `self_check_report`）。
  - Today/Dispatch 的只读规则性建议与 `waiting`/`nextActions` 的计算。
  - 多轮修订的记录与显示（audit trail）。
- V1 不实现：
  - 自动化批量替代律师审批或自动定稿
  - 将 AI 输出视为法律意见代替律师判断
  - 改变数据库 Schema 或新增持久化生命周期表

## 15. V2 扩展方向

- V2 可加入（需严格审计与权限控制）：
  - 授权式半自动执行：律师授权后，AI 在限定范围内批量执行建议（需可回滚）
  - 更细粒度的 AI 自检和可解释性输出（链路到证据来源）
  - ML 驱动的修订优先级排序与质量评分
  - 客户/第三方协作流（外部上传、审阅回合）

## 16. 验收标准

P2.3 验收应满足：

- 文档明确律师/AI 的职责边界与不可越权行为。
- 定义了律师必须确认的节点与审计要求。
- 描述了审核单轮/多轮流程与补充资料的再加工路径。
- 明确 UI/Today/Matter 的呈现方式与必要元数据字段。
- 给出至少三项示例场景与预期状态流（Documents/Evidence/Research）。
